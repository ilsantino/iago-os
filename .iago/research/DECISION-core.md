## Decision 2: Context Persistence

**Verdict:** ECC's event structure + Ruflo's data quality. No Autopilot.

Ruflo's "archive on every UserPromptSubmit" is a 1,979-line beast that solves a problem we don't have. On 200K context, sessions run long before compaction. Archiving every prompt is continuous I/O for marginal benefit. The real insight from Ruflo isn't the frequency — it's reading actual token usage from transcript JSONL instead of guessing. We take that and nothing else.

ECC's session trio (start, compact, stop) is the right event model. Three hooks, three clear responsibilities, easy to reason about. We add GSD's HANDOFF.json for structured pause/resume because consultancy work constantly gets interrupted — client calls, context switches, end of day.

**Hook events (4):**

| Event | What it does | Timeout |
|-------|-------------|---------|
| `SessionStart` | Loads most recent session snapshot + HANDOFF.json if present. Injects context via `hookSpecificOutput`. Deletes HANDOFF.json after load. | 5s |
| `PreCompact` | Reads transcript JSONL, extracts key state (files, decisions, task, tools). Writes session snapshot to `.iago/state/sessions/{id}.json`. Outputs compact instructions via `hookSpecificOutput` telling Claude what to preserve. | 15s |
| `Stop` | Writes final session summary — marks session complete with outcome, duration, token totals. Updates snapshot with `end_time` and `outcome: "completed"`. | 10s |
| `SubagentSessionEnd`* | *Not wired initially.* Future hook for when we spawn subagents — track their cost separately. |  |

Skipped events:
- **UserPromptSubmit**: Statusline already fires here for context % (Decision 8). Double-hooking this event adds latency to every prompt for archiving we don't need at 200K.
- **SessionEnd**: Fires on `/clear` and session end. Redundant with Stop for our purposes — Stop fires on all session exits.

**Data persisted (session snapshot schema):**

```json
{
  "session_id": "abc123",
  "start_time": "2026-03-31T14:00:00Z",
  "end_time": "2026-03-31T16:30:00Z",
  "outcome": "completed|paused|interrupted",
  "client": "acme",
  "project": "iago-os",
  "git_branch": "feat/auth",
  "compaction_count": 0,
  "files_modified": ["src/auth.ts", "src/login.tsx"],
  "files_read": ["src/db.ts", "package.json"],
  "tools_used": { "Edit": 12, "Read": 8, "Bash": 5 },
  "key_decisions": [
    "Chose JWT over session cookies for auth",
    "Using bcrypt for password hashing"
  ],
  "current_task": "Implement login flow with email/password",
  "total_tokens": { "input": 45000, "output": 12000 },
  "last_compaction": "2026-03-31T15:00:00Z"
}
```

**Key decisions extraction method:** Scan transcript JSONL for assistant messages containing decision markers: "decided", "choosing", "going with", "approach:", "verdict:", "we'll use", "picked". Extract the sentence containing the marker. Cap at 10 decisions per session. Crude but effective — same keyword-matching approach Ruflo uses, proven to surface the right content.

**Format:** JSON. One file per session. Not JSONL — session snapshots get updated multiple times (PreCompact writes, Stop finalizes), and JSONL is append-only. JSON lets us read-modify-write cleanly.

**Storage:** `.iago/state/sessions/{session-id}.json` — gitignored. Sessions are runtime state, not project artifacts. Keep last 10 sessions, prune older ones on SessionStart.

**HANDOFF.json:** Yes. Simplified from GSD.

When context hits CRITICAL (90%, per Decision 8) or the operator explicitly pauses, write `.iago/state/HANDOFF.json`:

```json
{
  "paused_at": "2026-03-31T15:30:00Z",
  "session_id": "abc123",
  "client": "acme",
  "current_task": "Implement login flow — password validation done, need OAuth next",
  "completed_steps": [
    "Email/password registration endpoint",
    "Login endpoint with JWT"
  ],
  "remaining_steps": [
    "OAuth Google provider",
    "Refresh token rotation"
  ],
  "blockers": [],
  "key_decisions": ["JWT over session cookies", "bcrypt for hashing"],
  "uncommitted_files": ["src/auth.ts", "src/routes/login.ts"],
  "next_action": "Implement Google OAuth — start with passport-google-oauth20 setup"
}
```

HANDOFF.json is NOT auto-generated by hooks. It's created by a slash command (`/iago:pause`) that reads the current session snapshot and asks Claude to structure the handoff. Hooks just read it on SessionStart and delete it after loading.

**Recovery flow:**

- **After compaction:** PreCompact already ran → session snapshot saved → Claude's compaction receives our `hookSpecificOutput` listing files, decisions, current task. Post-compaction Claude has a compressed but accurate picture. No hook fires post-compaction — the output we injected pre-compaction does the work.
- **After restart (`/clear` or new terminal):** SessionStart loads most recent session snapshot from `.iago/state/sessions/`. If HANDOFF.json exists, loads that instead (it's more structured and intentional). Injects via `hookSpecificOutput`: "Previous session was working on X, modified these files, made these decisions."
- **After crash (kill -9, power loss):** Stop hook never ran → session snapshot has no `end_time`. SessionStart detects this: `outcome` missing or no `end_time` = interrupted session. Loads the PreCompact snapshot (last known good state). User gets: "Previous session ended unexpectedly. Last known state: ..."

**Files created:**

| File | Lines | Purpose |
|------|-------|---------|
| `context-persistence.mjs` | ~250-300 | Single file, CLI arg dispatch (`session-start`, `pre-compact`, `stop`) |
| `lib/transcript.mjs` | ~80 | Transcript JSONL reader — shared between persistence + cost tracker + statusline |

The single-file-with-arg-dispatch pattern (Ruflo's approach) is deliberate here. These three events share state (session data, transcript parsing) so splitting them into three files would mean three copies of the same imports and session-file logic. One file, three code paths.

## Decision 6: Compaction Strategy

**Verdict:** Token-percentage only. No tool-call counter.

ECC's tool-call counter (suggest /compact at 50 calls) is a proxy metric. We have the real thing: actual token usage from transcript JSONL, computed by the statusline and written to bridge-ctx.json every render cycle (Decision 8). Using both would mean two systems trying to tell you the same thing with different accuracy. Kill the proxy.

**Mechanism:** Already architected in Decisions 2 + 8. No new hook needed.

1. `statusline.mjs` computes context % from transcript JSONL → writes `bridge-ctx.json`
2. `context-monitor.mjs` reads `bridge-ctx.json` on every PostToolUse → injects warnings at thresholds
3. `context-persistence.mjs pre-compact` fires when compaction actually happens → saves session state

The compaction *decision* (when to compact) is surfaced to the user via warnings. The compaction *execution* is Claude Code's built-in `/compact`. We don't auto-compact — the user or Claude decides when. Our job is to make sure they have good information and that nothing is lost when it happens.

**Thresholds (from Decision 8, confirmed):**

| Context used | Level | Action |
|-------------|-------|--------|
| < 65% | Normal | Green indicator in statusline |
| >= 65% | Advisory | Yellow indicator in statusline. No injection. |
| >= 80% | WARNING | Inject into conversation: "Context limited — finish current task, then compact or pause." Debounce: 5 tool uses between repeated warnings. |
| >= 90% | CRITICAL | Inject into conversation: "Context nearly exhausted. Run /compact now." No debounce — every PostToolUse reminds. Also suggest `/iago:pause` to write HANDOFF.json. |

**Debounce:** 5-tool-use gap between WARNING injections (GSD's approach). Prevents nagging. CRITICAL bypasses debounce — at 90%, every reminder is warranted. Debounce state tracked in bridge-ctx.json (`last_warning_tool_count` field).

**Preserved on compaction:**
- Session snapshot (files modified, files read, key decisions, current task, tools used, token counts, git branch, client slug) — written by PreCompact hook
- Compact instructions injected via `hookSpecificOutput` — Claude's compaction summary includes this

**Compact instruction format (what PreCompact injects):**
```
## Session Context (preserved by iaGO)
**Client:** acme
**Task:** Implement login flow with email/password
**Branch:** feat/auth
**Files modified:** src/auth.ts, src/routes/login.ts
**Key decisions:**
- JWT over session cookies
- bcrypt for password hashing
**Progress:** Registration endpoint done, login endpoint done, OAuth next
```

This is plain text, not JSON. Claude's compaction system works best with natural language context that it can fold into its summary.

**Discarded on compaction:**
- Full conversation turns (Claude's compaction handles this — it summarizes)
- Intermediate file reads (can re-read files)
- Tool output details (the summary captures what matters)
- Earlier session snapshots (only current session matters post-compaction)

**Integration with persistence:** Direct. PreCompact is one of the three code paths in `context-persistence.mjs`. It reads the transcript, builds the snapshot, writes it to `.iago/state/sessions/{id}.json`, and outputs compact instructions. Same file, same session state, no coordination needed.

**Files created:** None beyond what Decision 2 specifies. Compaction is handled by `context-persistence.mjs pre-compact` + `context-monitor.mjs` (already in the build).

## Decision 5: Cost Tracking

**Verdict:** Per-session utilization logging. Not per-token dollar costs.

On Claude Max 200, there's no per-token billing — it's a flat subscription. ECC's model-pricing table ($/1M tokens for Haiku/Sonnet/Opus) doesn't apply. But utilization tracking still matters for the consultancy: how many tokens is each client consuming? How many sessions? If one client is eating 80% of the team's Claude capacity, that affects staffing and pricing decisions.

If the team later moves to API billing, we add a pricing table then. For now: tokens, not dollars.

**Storage:** `.iago/state/costs.jsonl` — per-project, gitignored. Each project maps to a client (via `active-client.json`), so per-project storage IS per-client storage. For cross-project rollups, `cat ~/.iago/*/costs.jsonl` or a future summary script. No dual-write, no global file. Start simple.

**Format:** JSONL. Append-only, one line per session. Perfect for logging — no read-modify-write, no corruption risk from concurrent writes, `grep "acme" costs.jsonl | jq` for ad-hoc queries.

**Fields (per entry):**

```json
{
  "timestamp": "2026-03-31T16:30:00Z",
  "session_id": "abc123",
  "client": "acme",
  "project": "iago-os",
  "model": "claude-sonnet-4-6",
  "input_tokens": 45000,
  "output_tokens": 12000,
  "cache_read_tokens": 8000,
  "cache_creation_tokens": 3000,
  "session_duration_ms": 9000000,
  "compaction_count": 1,
  "git_branch": "feat/auth",
  "tools_used": { "Edit": 12, "Read": 8, "Bash": 5 },
  "files_modified_count": 4,
  "operator": "santiago"
}
```

Field rationale:
- `cache_read_tokens` / `cache_creation_tokens`: Available in transcript JSONL `usage` field. Relevant for understanding actual API load vs cache hits. Costs nothing to capture.
- `session_duration_ms`: Time from SessionStart to Stop. Correlates token usage with wall-clock time per client.
- `compaction_count`: How many times context was compacted. High counts = complex sessions = potentially underpriced client.
- `operator`: Who ran the session. Read from `$USER` / `$USERNAME` env var. Three people, need to know who worked on what.
- No `estimated_cost` field. Flat-rate subscription means any number we put here would be a lie.

**Client tagging mechanism:** Read from `.iago/state/active-client.json` — same file statusline reads (Decision 8). This file is set by the operator before starting work:

```json
{ "client": "acme", "project": "widget-redesign" }
```

For now, set it manually: `echo '{"client":"acme","project":"widget-redesign"}' > .iago/state/active-client.json`. A `/iago:client acme` slash command comes later to make this ergonomic. The file is simple enough that manual creation is fine for 3 people while we bootstrap.

If `active-client.json` is missing, `client` defaults to `"internal"` and `project` defaults to the git repo name. No error, no nag — not every session is billable.

**Hook event:** Stop. One entry per session. The Stop code path in `context-persistence.mjs` already reads the transcript JSONL for final token counts (Decision 2). Cost logging is 15 extra lines in the same Stop handler — read tokens from transcript, read client from `active-client.json`, append JSONL line. No separate hook file needed.

**File:** No new file. Cost tracking is ~15 lines added to the Stop path of `context-persistence.mjs`. The shared `lib/transcript.mjs` (from Decision 2) handles the transcript JSONL parsing including token extraction.

**Cost summary:** Not now. Raw JSONL is queryable with `jq` for a 3-person team:
```bash
# Total tokens per client this week
cat .iago/state/costs.jsonl | jq -s 'group_by(.client) | map({client: .[0].client, total: map(.input_tokens + .output_tokens) | add})'

# Sessions per operator
cat .iago/state/costs.jsonl | jq -s 'group_by(.operator) | map({who: .[0].operator, sessions: length})'
```

A `/iago:costs` summary command comes later when we have enough data to know what views matter.

## Dependency Map for Phase 4-5

### What Phase 3 decisions constrain:

- **Post-edit hooks (Phase 4)** add to the PostToolUse array with `"matcher": "Edit"`. They coexist with context-monitor (which has no matcher = fires on all tools).
- **Config protection (Phase 4)** is PreToolUse with `"matcher": "Write|Edit"`. No conflict with existing hooks.
- **Slash commands (Phase 5)** like `/iago:pause` write HANDOFF.json by reading the current session snapshot. They depend on the session snapshot schema defined above.
- **`lib/transcript.mjs`** is shared across statusline (context %), persistence (session data), and cost tracking (token counts). It's the single most critical utility — if it breaks, three hooks break.

### Open questions for Phase 4:
- **Decision 3**: Post-edit quality pipeline — which checks, which order?
- **Decision 4**: Config protection — what files are protected?
- **Decision 9**: Security guards — secret detection, prompt injection prevention?

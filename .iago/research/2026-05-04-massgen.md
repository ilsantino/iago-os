# Research: massgen/massgen

**Date:** 2026-05-04
**Repo:** https://github.com/massgen/MassGen

## What it is

MassGen (Multi-Agent Scaling System) is a Python framework that runs N agents in parallel on the same task, lets them observe each other's working outputs in real time via a `BroadcastChannel`, then converges via voting to select the best answer. It is emphatically NOT a pipeline or workflow DAG — it is a redundancy-and-refinement system: every agent works the whole problem, they cross-pollinate, and the best collectively-validated solution wins. Think competitive deliberation, not assembly line.

## Architecture pattern

**Parallel peer-agent collaboration with convergence voting.**

- `Orchestrator` class wraps N `SubAgent` instances. Agents are passed in at init, not spawned on demand.
- Each agent has isolated `AgentState` (answers, votes, injection counts, timeout tracking).
- `BroadcastChannel` enables direct agent-to-agent communication without orchestrator mediation.
- Agents call `workflow_tools` (vote, new_answer, broadcast, checklist submission) to signal decisions.
- `CoordinationTracker` logs voting history and answer revisions (anonymized agent mappings).
- Multiple convergence strategies: majority vote, checklist-gated, decomposition mode (subproblems), fairness gates.
- When any agent submits `new_answer`, all agents soft-restart with the updated context — continuous refinement, not one-shot.
- Per-round workspace isolation via snapshots prevents cross-agent state contamination.
- Supports 15+ providers: Anthropic, OpenAI, Gemini, Grok, Azure, Groq, Together, OpenRouter, vLLM, LM Studio.
- MCP integration built in — agents get tools via MCP servers.

Key distinction: iago-os hub-and-spoke means agents never talk to each other. MassGen's `BroadcastChannel` is the core architectural differentiator — lateral agent communication is the entire point.

## Patterns worth absorbing

1. **`BroadcastChannel` for lateral agent communication** — agents sharing working summaries mid-task without orchestrator mediation. Directly upgradeable for /council: instead of advisors writing isolated analyses, they could react to each other's drafts before synthesis. The peer-review step in /council is a manual prompt approximation of this; MassGen does it natively in async.

2. **`new_answer` soft-restart protocol** — when any agent produces a breakthrough, all agents receive it and restart their current reasoning. This prevents the common multi-agent failure mode where a superior answer exists but other agents continue pursuing inferior paths. Could inform how iago-os handles mid-pipeline breakthroughs (e.g., if the review agent finds a Critical finding, the impl agent restarts rather than continuing).

3. **`CoordinationTracker` with anonymized agent mappings + voting history** — structured observability over multi-agent deliberation. iago-os has no equivalent; /council synthesis is black-box from the advisor perspective. Absorbing this pattern means decisions become auditable: why did the council pick option A? Which advisors voted, and did any flip? Useful for strategic decisions where Santiago wants the reasoning trail, not just the verdict.

## Integration cost

**Estimate:** large

**What it would take:**
- MassGen is Python 3.11+; iago-os is TypeScript/Node first. No native drop-in.
- Running MassGen as a subprocess or via its LiteLLM-compatible API endpoint is feasible but adds an out-of-process dependency with its own config surface (YAML agent definitions, Docker optional, provider API keys).
- The Web UI + async runtime are baked into MassGen's architecture — you'd be running the full stack even to use just the orchestration logic.
- Pattern absorption (options 1-3 above) requires no dependency — just reimplementing the ideas in the iago-os agent system. That's the realistic integration path.
- Replacing /council with MassGen would require: standing up MassGen as a local service, writing a `/council-massgen` skill that POST-s a query and streams results, handling the TUI vs. text output mismatch, and deciding which agents/models map to which MassGen agent slots. Realistic estimate: 2-3 days to wire up + ongoing maintenance of a Python service alongside the Node stack.

## Verdict

**Recommendation:** clear-no (as a dependency) / clear-yes (as pattern source)

**Reasoning:** MassGen solves the same problem as /council but at runtime scale — it is genuinely the productized version of the Karpathy LLM Council pattern. The architecture is sound, license is clean, and maintainership is strong. However, introducing a Python 3.11 async service with Textual/Docker dependencies as a runtime component of iago-os adds friction disproportionate to the gain, especially since iago-os's /council use cases are infrequent strategic decisions, not throughput-critical workloads where MassGen's parallel speed (4x claimed) matters.

**If clear-no (dependency):** Do not add MassGen as a runtime dependency or subprocess.

**Pattern absorption (do this instead):**
- Absorb `BroadcastChannel` concept: upgrade `/council` so advisors receive a stripped summary of peer drafts before finalizing (one extra prompt round). Keeps everything in-process, no new deps.
- Absorb `CoordinationTracker`: add a `## Voting record` section to /council output showing which advisors agreed, disagreed, and flipped — makes synthesis auditable.
- Absorb `new_answer` restart: in `execute-pipeline.sh`, if review stage finds Critical findings, flag the impl session to hard-restart the relevant task rather than accumulating diffs on a broken foundation.

Monitor at ~2k stars for maturity signal. If they ship a stable Python SDK with clean programmatic API (no TUI required), reassess the wiring cost.

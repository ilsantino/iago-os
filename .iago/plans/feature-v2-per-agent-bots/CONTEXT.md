## Scope (what this ships)
1. **Per-agent bot runtime.** Generalize `runtime/telegram/` from one global poller to **one poller per standing agent**, each on its own token (resolved per-agent via `LoadCredential`/config). Each agent's poller routes only that agent's messages; `ALLOWED_USER` allowlist enforced per bot (private-DM only — the private-chat gate stays CLOSED). The existing approval-bus handshake + per-agent file-bus tagging are reused.
2. **Chief/broadcast bot.** ONE chief/orchestrator bot is the front door for: (a) ephemeral/short-lived workers that do NOT get their own bot, and (b) broadcast / "talk to all agents". Ephemeral workers report through the chief bot.
3. **`role` + `lifecycle` + `telegram` fields in agent config.** `role:"chief"|"worker"` (existing decision), `lifecycle:"standing"|"ephemeral"` (standing → own bot; ephemeral → via chief), `telegram:{ mode:"own-bot"|"via-chief"|"none", tokenRef?, allowedUser? }`.
4. **Provisioning UX wired into agent creation (Santiago direction 2026-05-30).** When an agent is created (CLI `add-agent` flow now; dashboard form in Phase 6), offer "Give this agent its own Telegram bot?":
   - **Guided mode (default):** prompt the operator to create the bot via @BotFather (`/newbot` → name `iago-<agent>` → copy token), paste the token + Telegram user-id; the daemon then AUTO-writes the token to the agent's `LoadCredential` secret file, binds agent↔bot, sets `ALLOWED_USER`, and starts the poller. (BotFather has NO create-API — the token mint is inherently a Telegram-side step; everything after the paste is automatic.)
   - **Pool mode (optional, zero-friction):** a one-time pre-minted bot pool (`telegram-bot-pool.json` — N tokens created via BotFather upfront); agent creation AUTO-ASSIGNS a free bot from the pool, no per-agent BotFather step. Pool exhaustion → fall back to guided mode + warn.
   - Either mode: `rotate-telegram-bot.sh` extended to rotate per-agent tokens; deprovision returns a pool token / revokes a guided token reference.
5. **Dashboard affordance (Phase 6 — coordinate with `feature-v2-dashboard-comms-kanban-tabs`):** a per-agent "Connect Telegram" control surfacing the same guided/pool flow + showing bind status (own-bot / via-chief / none) + a "rotate token" action.

## Decided constraints (do not relitigate)
- **Per-agent bots for standing agents; chief bot for ephemeral + broadcast.** Per-agent bot TOKENS are now ACCEPTED (reverses 2026-05-29). No forum-topics (dropped — per-agent bots give the per-agent presence; private DMs keep the security gate closed).
- **HTTP-409 is per-token** — each agent polls its OWN token, so N independent pollers never collide. (Do NOT run two pollers on one token.)
- **Secrets via the existing per-agent systemd `LoadCredential=` model** — no plaintext tokens in config/logs; reuse `provision-credentials.sh` patterns.
- **BotFather has no create-API** — never claim fully-automatic bot creation; automate everything after the token exists. Pool mode pre-creates; guided mode pastes.
- **Pipeline preserved. No Docker, no Postgres.**
- **Garry-impressed completeness** — token-leak paths tested (no token in stderr/telemetry/trace), poller-crash restart, pool exhaustion, rotate/deprovision, allowlist enforcement.

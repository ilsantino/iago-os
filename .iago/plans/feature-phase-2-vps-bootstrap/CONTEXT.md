# `.iago/plans/feature-phase-2-vps-bootstrap/` — v2 Phase 2 VPS Bootstrap (MWP L2 Stage Contract)

**Stage:** iago-os v2 Phase 2 — daemon install on Hostinger VPS via FAST single-hour cutover; OpenClaw archive (encrypted, 30-day retention); Telegram bot rotation via BotFather `/revoke`; Anthropic 3-profile credential provisioning; LanceDB drop (MemPalace canonical); WhatsApp deauth via Meta Graph API; first-real-workflow = PR-triage agent.
**Layer:** L2 — stage contract.
**Token budget:** 200–500 tokens of contract content.
**Sibling stage:** `runtime/CONTEXT.md` (broader v2 daemon build umbrella covering Phase 0 → Phase 12).
**Preceding stage:** `.iago/plans/feature-v2-phase-1-daemon/CONTEXT.md` (Phase 1 — daemon skeleton local, acceptance gate landed in commit `4ee40ee`).

This stage produces 5 numbered plan files (`01-*.md` … `05-*.md`) that, when executed through the iaGO pipeline, ship Phase 2 of v2: the daemon running as a system-level systemd unit on the Hostinger VPS (srv1456441 / 187.77.135.32), with Telegram control surface rotated to a fresh BotFather token, OpenClaw stopped + archived (age-encrypted, 30-day retention via systemd timer), Anthropic 3-profile credentials provisioned via systemd `LoadCredentialEncrypted=`, WhatsApp deauthed at Meta, and a cron-fired PR-triage agent producing a daily Telegram summary. Acceptance gate is the 8-criterion checklist in § 10 of the Phase 2 delivery spec (`.iago/research/2026-05-16-phase-2-vps-bootstrap-spec.md`).

## Process

1. **Plan** — original 7 plan files (`01-*.md` … `07-*.md`) generated by `/iago-plan` 2026-05-17. **PRE-EMPTIVELY SPLIT 2026-05-18** per `.iago/decisions/2026-05-18-phase-2-split-and-dispatch.md` into 13 plan files (`01a/01b`, `02a/02b`, `03a/03b`, `04a/04b`, `05a/05b`, `06`, `07a/07b`). Rationale: Phase 1 max-turns ceiling hit 100% of the time on 6-8 task plans with extensive test additions; every Phase 2 plan except 06 fit that profile. Splits cap at 3-4 tasks per plan.
2. **Stress test** — every plan (split + original) carries an inline `## Stress Test` section with Critical/Important/Minor findings. Pipeline step 0 skips re-stress on plans that already carry that section. Split plans inherit findings from the original, scoped to the split's tasks.
3. **Approval gate** — Santiago authorized 2026-05-18 to GSD on the split + dispatch plan. The cutover runbook (Plan 03b) executes against PRODUCTION VPS state — Santiago at keyboard for cutover, but dispatch + dual-review of plans does not require per-plan approval.
4. **Execute** — `/iago-execute feature-phase-2-vps-bootstrap`. Full 8-stage pipeline per plan, dispatched in **6 rounds** (parallel across tracks, sequential within tracks; Santiago merges after each round to fast-forward main):
   - **Round 1** (parallel, no deps): `01a`, `02a`, `07a` — 3 PRs
   - **Round 2** (parallel, each deps its `a` counterpart merged): `01b` (deps 01a), `02b` (deps 01a, 02a), `07b` (deps 07a) — 3 PRs
   - **Round 3** (parallel, deps Round 1+2 merged): `03a` (deps 01a, 01b, 02a, 02b), `04a` (deps 01a, 01b) — 2 PRs
   - **Round 4** (parallel, deps Round 3 merged): `03b` (deps 03a), `04b` (deps 04a, 03b, 07a, 07b) — 2 PRs
   - **Round 5** (parallel, deps Round 4 merged): `06` (deps 01b, 03b), `05a` (deps 03b, 04b) — 2 PRs
   - **Round 6** (final): `05b` (deps 05a) — 1 PR

   Total: 13 PRs. Per-round wall clock: ~1.5-2.5 hr (pipeline parallel + dual review parallel + merge). Total wall clock: ~9-15 hr spread across 2-4 days.
5. **Verify** — Phase 2 acceptance criteria 1–8 per spec § 10. Self-evidence (criterion #8) means real terminal log of the cutover + rollback + Telegram screenshot from phone embedded in PR description. NOTE: no staging VPS per Santiago override (spec § 10 criterion #3 proposed $9/mo staging KVM, Santiago skipped); cutover dry-run is executed locally via `runtime/scripts/test-cutover.mjs` (03a Task 3) before the real cutover window. Plan 05b's `phase-2-vps.test.ts` provides opt-in Tailscale-SSH e2e checks (IAGO_VPS_E2E=1) for routine + post-handoff verification.
6. **Cutover execution** — Plan 03a + 03b artifacts executed by Santiago at the scheduled window: **Sunday 2026-05-25 8pm US/Mexico** (deferred from original 2026-05-18 default per `.iago/decisions/2026-05-18-phase-2-split-and-dispatch.md`; 7-day defer gives realistic dispatch + dual-review cycle). Not part of the pipeline itself; pipeline ships the scripts + docs, Santiago triggers the cutover with the scripts present and tested.
7. **Session digest** — write `sessions/2026-MM-DD-iago-os-v2-phase-2.md` to the Obsidian vault per `~/.claude/rules/obsidian.md`. Update STATE.md per project rule.

## Decided constraints (do not relitigate during planning)

These bindings are locked by Santiago override 2026-05-16, the Phase 2 delivery spec, and existing memory entries. Any plan contradicting them is wrong.

- **FAST cutover** — NO "alongside OpenClaw" parallel-run period. Stop OpenClaw + install v2 same hour. Rollback path = restart OpenClaw from archive + re-rotate Telegram token.
- **Same Telegram bot via BotFather `/revoke`** — preserves `@handle` + chat IDs + allowed-user-ID semantics. Per-bot rotation rejected (spec § 3 Option A verdict).
- **`User=iago` dedicated systemd system user** — NOT ilsantino, NOT root. Provisioned via `useradd --system --no-create-home --shell /usr/sbin/nologin`. Keeps `ProtectHome=true` meaningful (spec § 1 verdict).
- **systemd `LoadCredentialEncrypted=`** for all secrets (Telegram + 3 Anthropic profiles + future webhook secrets). 1Password CLI provisioning-only; plaintext NEVER touches local or remote disk. Bridge to `process.env` via `runtime/daemon/cred-bootstrap.ts` before `loadConfig()` (spec § 1 + § 2).
- **LanceDB drops at cutover** — encrypted OpenClaw archive preserves the 72 KiB content for 30 days; MemPalace is canonical going forward (spec § 6 verdict). No active fact-extraction.
- **WhatsApp deauth at cutover** — Meta Graph API `DELETE /<WABA_ID>/subscribed_apps` + `DELETE /me/permissions` per spec § 7. NOT scripted-and-tested in CI (Meta-side creds + one-time op); lives as runbook + thin wrapper script that takes the IDs as args.
- **First-real-workflow Phase 2 = PR-triage agent** (Shape 1 PTY adapter; daily cron at 14:00 UTC = 09:00 EST; queries `gh pr list` across iago-os org repos; classifies + posts single Telegram summary). NOT Sentria port, NOT full pipeline-on-VPS migration (spec extension via migration-scope § 1).
- **No staging VPS** — Santiago skipped per FAST preference. Cutover dry-run via `runtime/scripts/test-cutover.mjs` (local Vitest harness with all systemd/tailscale/age commands stubbed), NOT a separate Hostinger KVM.
- **Anthropic profiles: PROVISION at Phase 2, ACTIVATE at Phase 3** — Plan 01 adds `authProfile?: "default" | "ilsantino" | "iaguito"` field to `AgentConfig` + provisions all 3 credentials. Adapter wiring (passing the right env var to spawned process) is Phase 3 work. Phase 2 hello-world + PR-triage use the `default` profile only.
- **age-encrypted OpenClaw archive** — pubkey at `/etc/iago-os/santiago-age.pub`; private key stays on Santiago's Windows box; 30-day retention via systemd timer (NOT cron — VPS has no crontab per audit).
- **Telemetry continues** — Phase 1 NDJSON event emission extends with `daemon-start.runUnder: "systemd"` field + new `cred-bootstrap-loaded` event kind. Cutover scripts ALSO write one-line NDJSON to `/var/log/iago-os/cutover.ndjson` per major step.
- **No new MCP servers in Phase 2** — out of scope.
- **No Docker.** systemd on VPS directly. Match Phase 1 build artifacts (Node.js 20, ESM).
- **No Postgres.** Phase 2 does NOT wire SQLite cost ledger (that lands Phase 8).
- **Pipeline preserved.** `scripts/execute-pipeline.sh` runs unchanged; pipeline runs from Santiago's Windows box, not from the VPS (pipeline-on-VPS migration is Phase 11 per migration-scope § 11).
- **Garry-impressed completeness standard.** Every script idempotent; every command verified; every rollback step time-estimated; zero TODOs without tracked issue + date.
- **SIGHUP credential reload (OQ17)** — Plan 06 ships handler that re-invokes `loadSystemdCredentials()` on SIGHUP.

## Summary

No-ship: the approval path lets command-supplied IDs escape the approvals directory, and successful approval claims can be lost permanently on partial failure.

## Findings

### P0 (Critical)

- **runtime/telegram/approval-bus.ts:210-214 — Unvalidated approvalId allows state-root path traversal and file corruption** [critical]
  `resolveApproval` builds filesystem paths directly from `approvalId` and then unlinks the computed pending path. Because Telegram text-form approvals accept arbitrary IDs, an allowed Telegram user can submit an ID containing `../` or path separators, causing this code to delete and then overwrite files outside `approvals/*` but still under the daemon state root, such as agent state JSON. This is a trust-boundary bug in the remote control surface, not just malformed local input.
  - **Action:** Validate `approvalId` at every public approval-bus entry point before path construction. Since IDs are generated with `crypto.randomUUID()`, reject anything that is not an exact UUID, or at minimum use `assertSafeIdentifier` plus a resolved-path containment check. Add tests for `/approve ../../agents/foo allow` and callback IDs with path separators.

### P1 (Important)

- **runtime/telegram/approval-bus.ts:214-256 — Approval resolution deletes the only pending record before durable publish** [high]
  The claim point is `unlink(pendingPath)`, but the resolved decision is only written later. If the process crashes, disk fills, permissions fail, or `atomicRename` throws after the unlink succeeds, the approval is neither pending nor resolved. `waitForApproval` will time out and `listPendingApprovals` cannot detect the ghost because the pending file has already been removed. This violates the file-bus durability invariant and can strand or misclassify human approval decisions under partial failure.
  - **Action:** Make resolution recoverable: acquire an exclusive claim/lock or atomically rename pending to a claimed/inflight path, durably write the resolved decision, then remove pending/inflight only after the resolved file is committed. On recovery, preserve enough state to retry or surface inflight approvals instead of losing them.

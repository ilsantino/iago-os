## Verdict

**REQUEST_CHANGES** — 3 CRITICAL, 4 IMPORTANT, 5 MINOR, 3 FORWARD. The three CRITICALs are correctness defects that survived stress test + plan-compliance review + Codex pass: (1) `persistAgentConfig` lacks the duplicate-agentId fail-loud guard that `resolveAgentOrg` was supposed to enforce (PR4 contract is half-honored); (2) cascading shutdown of children mutates the parent's child-set via `parentChildren.delete(parentId)` BEFORE iterating, but `shutdownAgent(childId)` re-enters and calls `cascadeShutdownChildren(childId)` for grandchildren — fine alone, but `shutdownAgent` ALSO writes the graceful marker for the child even when the cascade reason is a crash/restart of the parent (marker reason lies about why the child stopped); (3) `restartAgent` does NOT respect the generation-token discipline mandated by the Shape 2 ADR section — there is no provision for in-flight stale-response discard, and the `freshHandle` re-spawn returns a NEW handle id from `runtime.spawn`, but the `restartingPromises` map keys on the ORIGINAL handle id while the new handle is now in `this.handles` under a different id — concurrent restart callers receive a handle pointing to the old id, but `getHandle(oldId)` returns `undefined`.

## Findings

### [CRITICAL] runtime/daemon/agent-manager.ts:890-912 — `persistAgentConfig` does not enforce PR4 "agentIds globally unique" invariant at write time

**Why:** Stress-test PR4 says agentIds MUST be globally unique and `resolveAgentOrg` throws on duplicates. But `persistAgentConfig` writes `<handleId>.json` with no read-back-check for existing agentId in any other org's record. Two parallel `registerAgent({agentId: "alpha", org: "org-1"})` and `registerAgent({agentId: "alpha", org: "org-2"})` calls both succeed silently. The duplicate is only surfaced LATER when `resolveAgentOrg("alpha")` happens to be called — at which point both agents are already running, holding resources, and the failure mode is "ambiguous error from a lookup nobody expected to fail" instead of "second register call rejected at the trust boundary."

Same scenario on daemon restart: two `<handleId>.json` files with the same `agentId` in different orgs survive across boots. The first call to `resolveAgentOrg(agentId)` post-boot throws; the daemon now has two live recovered agents that cannot be resolved by id. The PR4 invariant is half-honored: read-side detection without write-side enforcement.

**Fix:** in `registerAgent`, BEFORE calling `runtime.spawn`, call `resolveAgentOrg(config.agentId)`; if it returns a non-null org different from `config.org`, throw `Error("Agent id already registered in org <existingOrg>: <agentId>")`. Also guard against concurrent registers via an in-flight `Set<agentId>` so two simultaneous `registerAgent` calls race-detect each other. Add a unit test: `registerAgent({agentId: "x", org: "a"})` then `registerAgent({agentId: "x", org: "b"})` throws on the second call.

### [CRITICAL] runtime/daemon/agent-manager.ts:230-251, 705-721 — `cascadeShutdownChildren` stamps `graceful` marker on children even when parent crashed/recycled

**Why:** `cascadeShutdownChildren` calls `this.shutdownAgent(childId, "SIGTERM")` for every child. `shutdownAgent` unconditionally calls `writeStopMarker(handleId, "graceful")` at line 238. So when a parent CRASHES (status callback fires `crashed` at line 700) or is force-RESTARTED (line 290 calls cascade before parent shutdown), every linked child gets a `graceful` marker on disk — lying about WHY it stopped.

The README's `.daemon-stop` semantics table is explicit: "graceful — Daemon stopped intentionally. Do NOT re-spawn on next boot." A child that was killed because its PARENT crashed is now treated as a clean shutdown on next boot. If `knownConfigs` includes the child, recovery skips the crash-replay branch and the child is silently lost. Combined with the H1 fix's no-marker-means-crash logic, the bug INVERTS: children get `graceful` markers and are skipped; only children whose registration was never persisted hit the crash-recovery path.

**Fix:** add a second-arg `cascadeReason: StopMarkerReason` to `cascadeShutdownChildren` and a private overload (or new method) `shutdownAgentInternal(handleId, signal, markerReason)`. From `handleStatusChange` "crashed" branch pass `"crash"`; from `doRestart` pass `"recycle"`; from the user-facing `shutdownAgent` path keep `"graceful"`. Update README. Add tests: parent crashes → child marker reason is `crash`; parent restart → child marker reason is `recycle`.

### [CRITICAL] runtime/daemon/agent-manager.ts:253-317 — `restartingPromises` key is stale after re-spawn; concurrent restart callers receive an unreachable handle

**Why:** `restartAgent(handleId, reason)` keys `restartingPromises` on the ORIGINAL handle id (line 268). The promise resolves to a `freshHandle` with `id = freshHandle.id` (whatever `runtime.spawn` returns — may or may not equal the original id). For the test mock, `spawn` increments a counter so `freshHandle.id = "mock-pty-h2"` while original was `"mock-pty-h1"`. The new handle is tracked under the NEW id (line 308 `trackHandle` calls `this.handles.set(handle.id, ...)`). Test passes because it only inspects the returned `restarted` handle.

Real callers: a second `restartAgent("mock-pty-h1", "stalled")` arriving during the in-flight restart gets the cached promise → receives `newGeneration` whose id is `"mock-pty-h2"`. They then call `manager.getHandle("mock-pty-h1")` → `undefined`. Or they call `restartAgent("mock-pty-h1", ...)` a third time AFTER the first resolves → `this.handles.get("mock-pty-h1")` is undefined → throws `"No handle to restart: mock-pty-h1"`. The heartbeat is also re-registered under the new id by `trackHandle` line 663, so the heartbeat's stale registration under `"mock-pty-h1"` is removed in `teardown` (line 923) but the in-flight `restartingPromises` map briefly holds a key that no longer maps to a live handle.

The mock test masks this by having `spawn` return predictable counter-suffixed ids — production PTY adapter (Plan 04) is likely to generate truly unique uuids per spawn, exposing the bug as soon as any operator calls `restartAgent` twice from two code paths.

**Fix:** either (a) require adapters to PRESERVE the handle id across restart (document on `AgentRuntime.spawn`), and verify in `doRestart` that `freshHandle.id === handleId` — throw if not; (b) thread an "alias map" `Map<oldHandleId, newHandleId>` so callers using the old id can still resolve through `getHandle`; or (c) document at the `restartAgent` contract that "the handle id MAY change; callers must use the return value." Option (a) matches the README claim "the new handle is keyed on its new id, and the old file is no longer referenced" (line 499 of the agent-manager comment about removeOrphanAgentConfig — the comment ALREADY admits the new id differs). If (a) is chosen, the ADR Shape-2 generation-token discipline applies: id stable, generation increments. Pick one and write a test that exercises the chosen contract.

### [IMPORTANT] runtime/daemon/agent-manager.ts:319-378 — `spawnSubagent` parent-alive race is checked AFTER spawn returns; expensive child is created then killed in the loss case

**Why:** EC2 was supposedly fixed by re-checking `parent.runtime.isAlive(parent.handle)` AFTER `runtime.spawn` returns. The race window is now smaller but still exists: between `spawn` returning a child handle and the `isAlive` check, the parent's status callback can fire `exited` and execute `cascadeShutdownChildren` — which finds the parent's child-set EMPTY (the new child hasn't been linked yet at line 375), returns silently, and the child orphans for a different reason than EC2 intended. The new check catches the case where parent died BEFORE spawn returned, not the case where parent died DURING/AFTER spawn but before linkage insertion.

Also: `runtime.spawn` may already be expensive (PTY allocates a real terminal). Cheap pre-check (`parent.runtime.isAlive(parent.handle)`) BEFORE `runtime.spawn` would short-circuit most cases without paying the spawn cost.

**Fix:** wrap the entire spawn+linkage block in a mutex keyed on `parentHandleId`. Sequence: (1) acquire parent-lock; (2) check `isAlive`; (3) call `runtime.spawn`; (4) re-check `isAlive` AND `this.handles.has(parentHandleId)`; (5) insert linkage; (6) release lock. The cascade path acquires the same lock when iterating children, so it cannot start tearing down children mid-spawn.

### [IMPORTANT] runtime/daemon/heartbeat.ts:94-99 — `start()` `setInterval` callback drops the previous `inflightSweep` reference; tick-faster-than-sweep loses the await chain

**Why:** `start()` sets `this.inflightSweep = this.sweep()` on every tick. If `intervalMs=60_000` and a sweep takes 2s, fine. But the production daemon is liable to set short intervals during recovery, debugging, or for fast-shape adapters. If a sweep is still running when the next tick fires, `this.inflightSweep` is OVERWRITTEN by the new `this.sweep()` promise — the previous in-flight sweep continues but is no longer awaitable from `stop()`. `stop()` only awaits the LATEST `inflightSweep` (line 108), so test teardown that calls `stop()` may leave the prior sweep running into the next test's beforeEach, corrupting temp-dir state.

The probe loop iterates `await probe()` and `await this.onForceRestart(...)` serially — a hanging probe (e.g., a stuck `runtime.getStatus` call without a timeout) stalls the entire sweep beyond `intervalMs`, guaranteeing overlap.

**Fix:** either (a) guard `start()`'s callback with `if (this.inflightSweep !== null) return; this.inflightSweep = this.sweep().finally(() => { this.inflightSweep = null; })` — skip the tick if the previous sweep is still running; or (b) track an array of in-flight sweeps and await all on stop. (a) is cheaper and matches "heartbeat ticks are sample-and-skip, not queued." Add a test: register a probe that resolves after 200ms with `intervalMs=50`; assert sweep count <= ticks (no overlap).

### [IMPORTANT] runtime/daemon/heartbeat.ts:135-181 — `sweep()` does not snapshot `this.handles` defensively against mid-sweep `unregister`

**Why:** Line 137 iterates `this.handles.entries()` into a `snapshot` array. Good — but the probe at line 144 calls `await probe()`, during which another async path (cascade shutdown, restart, IPC unregister) may call `this.unregister(handleId)`. The snapshot still iterates the stale entry, calls its probe, and may invoke `onForceRestart` on a handle the manager already discarded. The manager's `restartAgent` then throws `"No handle to restart"` (line 265), which is swallowed at line 187-194 — but the operator sees stderr spam about a restart that should never have been attempted.

**Fix:** in the sweep loop, re-check `this.handles.has(handleId)` BEFORE invoking `onForceRestart`. Skip with no log if the handle is gone — the unregister is the source of truth.

### [IMPORTANT] runtime/daemon/agent-manager.ts:776-835 — `attemptCrashReplay` does not validate `cfg.runtimeId` matches the adapter version that wrote the session log

**Why:** The replay re-feeds `prompt` and `inject` events through the CURRENT runtime resolved by `cfg.runtimeId`. If an operator upgraded the adapter between boots (e.g., `claude-pty` v0.0.1 → v0.0.2 with breaking `send` semantics), the replay sends old-format messages to a new-format adapter. EC4 in the stress test acknowledged the gap "document in README for Phase 3+." The README documents it (line 213) but the code has zero version check. A version-mismatch on boot silently corrupts replay.

For Phase 1 this is acceptable (only one adapter version exists), but the code should at least RECORD the runtime version observed at registration time in `persistAgentConfig`, and on replay compare against the current runtime's `version` field — log a warning when they differ.

**Fix:** add `runtimeVersion: runtime.version` to the persisted config; in `attemptCrashReplay`, after `resolveRuntime(cfg.runtimeId)`, compare `runtime.version === cfg.runtimeVersion`. On mismatch, log `[agent-manager] adapter version drifted on replay for ${handleId}: was=${cfg.runtimeVersion}, now=${runtime.version}` and continue. Update the README's "Failure modes" row for adapter-version mismatch to point to this warning.

### [MINOR] runtime/daemon/markers.ts:48-65 — `writeStopMarker` uses direct write, not write-temp-then-rename

**Why:** Probe Q4 — power loss during marker write leaves the file in a partial state. Current code opens `"w"`, calls `handle.writeFile(JSON.stringify(marker))` then `handle.datasync()`, then closes. If the process is hard-killed (e.g., OOM kill) between truncation and the writeFile completing, the file exists but is empty or truncated → `readStopMarker` JSON.parse fails → treated as absent → next boot treats this as a CRASH even though it was a graceful shutdown that lost its marker mid-write.

Convention everywhere else in the daemon (per state-paths.ts comment): write to tmp then `atomicRename`. The Plan 02 contract for HWM does this; markers do not.

**Fix:** open `<handleId>.daemon-stop.tmp`, write + datasync, close, then `atomicRename(tmp, final)`. This guarantees the final file is either fully written (correct content) or absent (treated as crash — which is the safe default behavior of the recovery branch).

### [MINOR] runtime/daemon/agent-manager.ts:165-220 — `registerAgent` writes `persistAgentConfig` AFTER `trackHandle` (heartbeat wired); crash between spawn and persist loses recovery info

**Why:** Order: spawn (line 209) → trackHandle (line 210, wires heartbeat) → persistAgentConfig (line 218). If the process dies between trackHandle and persistAgentConfig, the spawned PTY exists in the OS process tree but no `<handleId>.json` was ever written. On next boot, `knownConfigs` excludes this id, the H1 "no-marker-means-crash" branch never fires for it, and the orphaned PTY child process leaks (worst case) or hits a SIGCHLD with no consumer.

**Fix:** write `persistAgentConfig` BEFORE calling `runtime.spawn`. If spawn fails, leave the config file in place — it points at a never-spawned id, but H1's recovery will see "no marker" and attempt `restoreFromMarker` which returns null → recorded as crash with no replay (correct outcome). Alternative: write a `pending` flag in the config and clear it after spawn returns. First option is simpler.

### [MINOR] runtime/daemon/agent-manager.ts:592-664 — `trackHandle` swallows `tracked.unsubscribe = ()=>{}` initial placeholder; if `runtime.onStatusChanged` throws synchronously, the placeholder leaks

**Why:** Lines 604, 619-622. The `tracked` record is inserted into `this.handles` at line 615 BEFORE `onStatusChanged` is called (line 619). If `runtime.onStatusChanged` throws (a broken adapter), the catch is missing — the throw propagates up through `registerAgent`, but `this.handles` retains the entry with the no-op unsubscribe placeholder. Subsequent `shutdownAgent(handleId)` calls the no-op unsubscribe (fine, swallowed) but the heartbeat was also already registered at line 663 — except line 663 is AFTER the `onStatusChanged` line, so the broken-adapter throw skips heartbeat registration too. Net: orphan entry in `this.handles` with no status subscription and no heartbeat probe.

**Fix:** wrap `runtime.onStatusChanged(...)` in try/catch; on throw, remove from `this.handles`, re-throw.

**Why:** Loop at line 710 awaits each `shutdownAgent` serially. If `runtime.shutdown` blocks for 30s (the documented timeout from Plan 04), a parent with N children takes 30N seconds to teardown. SIGTERM cascade should fan-out.

### [FORWARD] generation-token discipline NOT enforced for PTY (Shape 1) restart; ADR § Shape 2 contract leaks into a placeholder

**Why:** The ADR § Shape 2 explicitly mandates generation-token-based stale-response discard for HTTP/SDK callbacks. The current `restartAgent` increments `generationToken` (line 306) but no callsite ANYWHERE verifies the token before mutating state on a callback. For Shape 1 PTY, status callbacks come from a stdio reader that is tied to the child process — when the child dies and a new one spawns, the new pty has its own reader; stale callbacks from the old reader cannot arrive. So for Shape 1 the discipline is moot.

For Plan 04 (Shape 1 PTY adapter, PR #43 next in stack): confirm the PTY adapter does not retain a closure over a stale child's stdio. For Phase 3 (Shape 2 HTTP/SDK adapter): the daemon side MUST check `handle.generationToken === current.generationToken` before mutating cost/status from any AsyncIterable callback (`costTap`, `onStatusChanged`). The check belongs in `applyCostEvent` and `handleStatusChange` — neither currently consult generation.

**Fix (forward, not blocking):** add a TODO comment at `handleStatusChange` and `applyCostEvent`: "Phase 3 Shape 2/3: validate `handle.generationToken === tracked.handle.generationToken` before mutating; discard stale callbacks." Acceptable as a forward-list entry, but the comment should land in PR #42 to avoid losing the discipline in Phase 3 planning.

### [FORWARD] Windows process-tree cleanup of subagents not addressed; grandchildren orphan on parent kill

**Why:** Probe Q7. The cascade shutdown calls `runtime.shutdown(handle, "SIGTERM")` for each child. On Windows, `SIGTERM` is mapped to `process.kill` which on Node terminates ONLY the target process — child processes of that child are NOT in a job object by default, so they orphan. cortextOS uses `taskkill /T /F` on Windows (the memory note `feedback_worktree_hygiene.md` cites this for vite cleanup). The PTY adapter (Plan 04) needs to know to use `taskkill /T` or to wrap children in a job object via `winapi-job-object` or equivalent.

**Fix (forward):** add a section to runtime/daemon/README.md "Windows process-tree cleanup — adapter responsibility" pointing Plan 04 implementer at taskkill /T or job-object wrapping. Track as Plan 04 acceptance criterion.

### [FORWARD] Cost-tap on subagent that dies AFTER parent shutdown but BEFORE parent teardown — race window

**Why:** Probe Q2. `consumeCostTap` (line 723) iterates `for await (const event of runtime.costTap(handle))` — runs as a fire-and-forget `void` task. If the parent shuts down (cascade fires SIGTERM on child), the child's `costTap` may emit a final `CostEvent` BEFORE the iterator closes. `applyCostEvent` walks the parent chain (line 752); the parent is STILL in `this.handles` at that moment (cascade hasn't deleted it yet). The cost rolls up. Then parent shuts down and `this.handles.delete(parentId)` runs. Correct outcome.

But: if costTap emits AFTER `this.handles.delete(childId)` (line 925) — i.e., after `teardown(childId)` ran inside the cascade `shutdownAgent` call — the line `const tracked = this.handles.get(handleId); if (tracked === undefined) return;` (line 747-748) silently drops the event. **The drop is silent. No telemetry, no audit log.** Adversarial cost reconciliation against the Phase 8 SQLite ledger will show drift.

**Fix (forward):** when applying a cost event finds no tracked handle, emit a telemetry warning `[cost-tap-after-teardown] handleId=... amount=... — event dropped` so the drift is observable. Acceptable for Phase 1 but log the discrepancy.

## Cross-cutting probes

### Q1: Subagent shutdown ordering during cascading shutdown
Cascade fires children FIRST in both `shutdownAgent` (line 245) and `doRestart` (line 290) — correct. Marker semantics for the child are WRONG (Critical #2 above). Children in `claimed/` task files: NOT addressed — the file-bus has no concept of "owner died." Plan 02 file-bus assumes the claimer survives or fails-then-reclaims. **Recommendation:** Plan 04 must add an orphan-claim sweeper.

### Q2: Cost rollup timing
Eager on every event via `applyCostEvent`. Race with parent-teardown discussed in FORWARD #3. Race with parent-restart: if child cost arrives during parent's restart window, parent map entry is briefly absent (teardown ran, trackHandle hasn't yet) → rollup silently drops the event.

### Q3: Heartbeat 60s + 5min stall threshold vs. legitimate slow work
Plan 04 PTY adapter for `git clone` example: the adapter's `onStatusChanged` fires `running` once at spawn and then stays in `running` for the whole clone — `lastStatusChangeMs` does NOT refresh because status DOESN'T CHANGE (PR1 reset only happens on `onStatusChanged` callback, line 682). 5min `git clone` exceeds stallThreshold → force-restart misfire. **This is a real bug**, not theoretical. The PR1 fix is correctly implemented for cases where status fires periodically, but a long-running operation in steady `running` state never refreshes the timestamp.

**Promote to IMPORTANT #5:** stall detection must include a periodic-liveness signal independent of status changes. Either (a) the adapter emits periodic `running` callbacks at heartbeat cadence, or (b) the stall check uses `isAlive()` result as a tiebreaker — if alive AND no status change in stallThreshold, that's healthy steady-state work, not stall. Document the chosen convention.

### Q4: Marker write power-loss → addressed as MINOR #1 above.

### Q5: restartIfStalled idempotency → addressed by `restartingPromises` map; still vulnerable to the id-mismatch bug (CRITICAL #3).

### Q6: Generation token discipline → FORWARD #1 above.

### Q7: Windows process tree → FORWARD #2 above.

### Q8: Subagent identity collision → CRITICAL #1 (write-side enforcement missing).

### Q9: AgentManager.register idempotency → not idempotent. Same `(agentId, runtimeId)` twice triggers a SECOND `runtime.spawn` and overwrites `<handleId>.json` if handle ids happen to collide. The handle id collision is unlikely in practice (mock test uses a counter; real adapters use uuids), but the spec doesn't say "register is idempotent" so this is at least underdocumented. Document either way.

### Q10: Crash recovery from .daemon-stop → `bootRecovery` correctly attempts `restoreFromMarker` on crash markers AND on missing-marker-with-knownConfigs (H1 fix). Per-adapter `restoreFromMarker` returning null is handled (line 795). README contract documented.

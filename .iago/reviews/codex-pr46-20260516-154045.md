## Summary

No ship: the packaged daemon entrypoint does not start the daemon, and the advertised crash-recovery path is not wired into startup.

## Findings

### P0 (Critical)

- **runtime/daemon/main.ts:308-322 — Packaged daemon script exits without starting anything** [critical]
  `runtime/package.json` starts `dist/daemon/main.js`, but this module only exports `main()` and never invokes it when run directly. Import-based tests call `startDaemon()` and therefore miss the actual operator path. The documented/manual command will load the module, do no startup work, and exit without IPC, Telegram, heartbeat, or auto-started agents.
  - **Action:** Add a direct-execution guard that calls `main()` for the built module, and add an integration/smoke test that runs the compiled start command or spawns `dist/daemon/main.js` and observes daemon startup.

### P1 (Important)

- **runtime/daemon/main.ts:112-126 — Startup never supplies knownConfigs, so boot recovery cannot recover agents** [high]
  `startDaemon()` calls `agentManager.bootRecovery()` with no `knownConfigs`. In `AgentManager`, crash replay and recycle respawn both require that map; the daemon README also states the absent-marker recovery path only fires when the Phase 1 entry point supplies it. As wired here, persisted handles from a prior crash or recycle marker are only classified for telemetry and not restored/replayed by the real daemon startup path.
  - **Action:** Build and pass a `knownConfigs` map during startup, either from persisted handle configs plus current config/env data or from the resolved daemon config keyed by prior handle IDs, and cover crash-marker and absent-marker recovery through `startDaemon()` rather than only `AgentManager` unit tests.

### [CRITICAL] runtime/daemon/file-bus.ts:127–134 + 362–379 — Empty/malformed claim file strands task forever

`writeClaimDurably` opens via `wx` (creates 0-byte file via `O_CREAT|O_EXCL`) and THEN writes contents. If `handle.writeFile` fails (ENOSPC, EIO, SIGKILL mid-call), the empty `.claim.json` remains on disk because the `try/finally` only closes the handle — it never unlinks the partial.

Next `claimTask` call hits `EEXIST` on the 0-byte file. `readClaimFile` sees zero-byte content, `JSON.parse("")` throws, returns `null`. `claimTask` returns `{ claimed: false, reason: "already-claimed" }` (line 208) with NO `existingOwnerId`.

`reclaimIfStale` (line 362) reads the claim with `readClaimFile` → `null` → **returns `false` immediately, never unlinks**. The empty claim has no `claimedAt` so age-based cleanup cannot apply. **Task is permanently stranded** — no agent will ever pick it up, and no recovery mechanism in Plan 02 unblocks it.

  Why: combines two bugs — (a) crash-window between `wx` create and content write leaves orphan, (b) `reclaimIfStale` ignores malformed/empty claims. The exact 2026-05-14 Windows crash failure mode the stress test claimed E1 fixed.
  Fix:
  ```ts
  // file-bus.ts writeClaimDurably — unlink on any write/datasync failure:
  async function writeClaimDurably(claimPath, contents) {
    const handle = await fsp.open(claimPath, "wx");
    let wrote = false;
    try {
      await handle.writeFile(JSON.stringify(contents));
      await handle.datasync();
      wrote = true;
    } finally {
      await handle.close();
      if (!wrote) {
        await fsp.unlink(claimPath).catch(() => undefined);
      }
    }
  }

  // file-bus.ts reclaimIfStale — also clear malformed/empty claims:
  export async function reclaimIfStale(taskId, maxAgeMs) {
    assertSafeIdentifier(taskId, "taskId");
    const claimPath = claimPathOf(taskId);
    const claim = await readClaimFile(claimPath);
    if (claim === null) {
      // Could be missing (return false) OR malformed/empty (must reclaim).
      try {
        const st = await fsp.stat(claimPath);
        if (st.size === 0 || st.isFile()) { // malformed sentinel
          await fsp.unlink(claimPath);
          return true;
        }
      } catch (err) {
        if (getErrnoCode(err) === "ENOENT") return false;
        throw err;
      }
      return false;
    }
    // ...rest unchanged
  }
  ```
  Add a regression test: pre-plant a 0-byte `.claim.json`; assert `claimTask` and `reclaimIfStale(taskId, 0)` recover the slot.

### [CRITICAL] runtime/daemon/session-log.ts:90–117 vs 215 — Sequence semantics diverge: `seq` in replay counts ALL non-empty lines (incl. malformed); `loadSequence` recovery counts non-empty lines too — but `appendEvent`'s sequence is the **valid-line index**, not the **line-position index**

In `readEventsUpToHWM` (line 215): `seq++` BEFORE the `JSON.parse` try/catch, so malformed lines DO advance `seq`. The yielded sequence is the line-position ordinal.

In `loadSequence` (line 109–110): line count is line-position ordinal too — `for (const line of raw.split("\n")) { if (line.length > 0) lineCount++ }` counts ALL non-empty lines including malformed ones.

But `performAppend` (line 154): `nextSeq = currentSeq + 1` — **monotonic per successful write**. So if a daemon runs:
1. appendEvent → seq=1 (writes line 1)
2. someone hand-edits a malformed line at line 2 (or a partial write left one)
3. appendEvent → seq=2 (writes line 3 of the file)
4. crash, `.seq` lost
5. recovery: `loadSequence` counts 3 non-empty lines → returns 3
6. next appendEvent → seq=4 (writes line 4 of the file)

But step 3's line was logically `seq=2`, persisted as `.seq=2`. After recovery, `seq=4` would be allocated to the next line, **skipping `seq=3`** — but more importantly, if step 3 wrote `seq=2` and a downstream consumer recorded HWM=`{byteOffset: end_of_line_3, sequence: 2}`, then post-recovery `readEventsUpToHWM` uses line-position seq, **mismatch** between what `appendEvent` allocated and what `readEventsUpToHWM` reports.

Inside the replay loop (lines 215–217), both `cumulativeBytes > hwm.byteOffset` AND `seq > hwm.sequence` are break conditions. If line-position-seq diverges from append-allocated-seq, the second guard fires early and silently truncates replay.

  Why: replay can stop before the real HWM if any malformed line precedes it (or appears in the file at all). Quiet data loss on the replay path; no test catches this because tests never combine malformed-line + HWM-with-sequence-past-it + actual write sequence.
  Fix: pick one definition and enforce it. Recommend "sequence = ordinal among successfully-parsed events" (the meaningful one):
  ```ts
  // readEventsUpToHWM — increment seq ONLY on valid parse, not on every non-empty line
  while (position < raw.length) {
    const newlineIdx = raw.indexOf("\n", position);
    if (newlineIdx === -1) break;
    const line = raw.slice(position, newlineIdx);
    const lineBytes = Buffer.byteLength(line, "utf8") + 1;
    cumulativeBytes += lineBytes;
    position = newlineIdx + 1;
    if (line.length === 0) continue;
    if (cumulativeBytes > hwm.byteOffset) break;
    let event;
    try { event = JSON.parse(line); }
    catch { console.error(...); continue; }  // do NOT increment seq
    seq++;
    if (seq > hwm.sequence) break;
    yield { event, sequence: seq };
  }
  ```
  AND change `loadSequence` recovery to count successfully-parsed lines (or trust `.seq` strictly and require a separate "rebuild" tool to handle corruption — explicit ops, not silent recovery).

  Also update JSDoc line 34 ("sequence is the line ordinal, not the yield ordinal") — that comment documents the BUG.

### [CRITICAL] runtime/daemon/session-log.ts:170–182 — `pauseIntake` doesn't drain in-flight appends; events can land between pause and replay

`pauseIntake` (line 310–313) just sets `state.paused = true` and returns. An `appendEvent` call that already passed the paused check at line 176 is mid-flight inside `performAppend` — `await fsp.open(..., "a")`, `writeFile`, `datasync`, etc. take real wall time. After `pauseIntake` returns, those in-flight appends complete and write NEW lines to the log.

`replay()` then reads `getHWM()`, iterates events. The `replay` window can see events APPENDED AFTER pause. The contract from CONTEXT.md "live intake paused during replay" (binding constraint) is violated.

  Why: replay can observe and re-deliver events that were already past the HWM line, OR worse, callbacks fire for events that arrived during the replay window — non-deterministic recovery.
  Fix: `pauseIntake` must await the file lock to drain in-flight writes:
  ```ts
  async pauseIntake(): Promise<void> {
    const state = getPauseState(this.handleId);
    state.paused = true;
    // Drain any in-flight performAppend by acquiring the lock once and releasing.
    await withFileLock(this.handleId, async () => {});
  }
  ```
  Note: `withFileLock` is module-private — either export `_drainFileLockForTests` and a public `drainIntake`, or restructure.

  Add a test: pre-pause kicks `appendEvent` calls without `await`; pauseIntake; assert no further writes appear in the file beyond pre-pause count.

### [CRITICAL] runtime/daemon/session-log.ts:328–340 — `resumeIntake` race: new `appendEvent` calls during drain bypass the queue and may interleave with drained items

`resumeIntake` flips `paused=false` BEFORE draining. Lines 330 (`state.paused = false`) then loop on `state.queue.splice(0)`. Any NEW `appendEvent` call arriving during the drain loop hits `state.paused === false` → calls `performAppend` directly. That call enters `withFileLock` and queues behind the currently-draining `performAppend`. **Drain order is preserved** for the snapshotted queue, BUT new arrivals can interleave with later drain items.

Concrete failure: queue is [D, E, F, G, H]. resumeIntake flips paused=false. Drains D. New caller arrives → calls performAppend(X). withFileLock orders: drain-E, X, drain-F, drain-G, drain-H. File order: A B C (pre-pause) D E X F G H. Sequence allocated by performAppend: D=4, E=5, X=6, F=7, G=8, H=9. **The caller of `appendEvent({E})` got back `seq=5`, the caller of `appendEvent({F})` got back `seq=7`. The values are correct individually but the surface contract "during pause your appends queue in arrival order; after resume they drain in that order before new arrivals" is broken.**

  Why: the docstring contract in lines 22–25 ("Each caller's promise resolves only after the event is durably written, which happens during `resumeIntake`'s in-order drain") implies queued events finish before new ones. Code violates this. Causes nondeterministic interleaving on busy handles, undermining the replay-determinism guarantee.
  Fix: flip flag AFTER drain — but then new `appendEvent` callers see `paused=true` and re-queue. Need an atomic transition: set a sentinel `draining=true`, keep `paused=true` until drain complete, then flip both atomically inside the lock:
  ```ts
  async resumeIntake(): Promise<void> {
    const state = getPauseState(this.handleId);
    const drained = state.queue.splice(0);
    for (const item of drained) {
      try {
        const result = await performAppend(this.handleId, item.event);
        item.resolve(result);
      } catch (err) {
        item.reject(err);
      }
    }
    state.paused = false;  // unblock new arrivals only AFTER drain
  }
  ```
  This still has a tiny race (new caller checks `state.paused` at line 176, sees true, queues; then resumeIntake completes drain and flips flag without checking the queue again). Fix that too: re-drain after flip in a loop until queue empty AND flag flipped under same critical section. Or use a proper async semaphore.

### [CRITICAL] runtime/daemon/session-log.ts:148–168 — Sequence cache poisoning on crash between `sequenceCache.set` and `persistSequence`

Lines 164–165:
```ts
sequenceCache.set(handleId, nextSeq);
await persistSequence(handleId, nextSeq);
```

Cache updated before disk. If daemon crashes here:
- log file has new line written + fsync'd ✓
- `.seq` still holds `currentSeq` (old value)
- in-process cache reflects new value but process is dead

Restart: `loadSequence` reads `.seq` → currentSeq. Next append → `nextSeq = currentSeq + 1` — **duplicates the sequence already written to the log line**.

`.seq` recovery via line-count (line 106–116) only triggers if `.seq` is ABSENT, not stale. Result: 2 distinct events share the same sequence number in the log, breaking downstream "sequence is unique key" assumptions.

  Why: any process crash inside the 1–2ms window between `datasync` returning and `.seq` write committing leaves a permanent off-by-one. With multi-process operation (Plan 07 ipc-server may invoke this from daemon while another caller is mid-flight), the window matters.
  Fix: write `.seq` BEFORE the log write, then commit log. Sequence is "reserved before written":
  ```ts
  async function performAppend(handleId, event) {
    return withFileLock(handleId, async () => {
      const currentSeq = await loadSequence(handleId);
      const nextSeq = currentSeq + 1;
      // 1. reserve the seq durably FIRST
      await persistSequence(handleId, nextSeq);
      sequenceCache.set(handleId, nextSeq);
      // 2. then write the line
      const line = `${JSON.stringify(event)}\n`;
      const handle = await fsp.open(logPathOf(handleId), "a");
      try {
        await handle.writeFile(line);
        await handle.datasync();
      } finally {
        await handle.close();
      }
      const stat = await fsp.stat(logPathOf(handleId));
      return { byteOffset: stat.size, sequence: nextSeq };
    });
  }
  ```
  Now if the crash happens between `.seq=N+1` write and log write, recovery sees `.seq=N+1`, next append uses `seq=N+2`, line at `seq=N+1` was never written → gap in sequence. Downstream MUST tolerate gaps but never duplicates — gap is recoverable (skip-and-warn in replay), duplicate is unrecoverable corruption.

  Add a test: fake-crash by killing the process between datasync and persistSequence (simulate via mock that throws on persistSequence). Restart, assert next append produces non-duplicate seq.

### [CRITICAL] runtime/daemon/session-log.ts:170–180 — Queued appends never reject on shutdown

If daemon is shutting down with events in `state.queue`, the promises returned to callers never resolve OR reject. The `_resetSessionLogStateForTests` clears the map but production has no shutdown hook. Callers `await appendEvent(...)` hang forever.

  Why: Plan 07 daemon shutdown will call `process.exit` or signal handlers; any active `await appendEvent` mid-pause leaks promises and may prevent graceful exit.
  Fix: expose `cancelPendingAppends(handleId): void` that drains queue with rejections:
  ```ts
  export function cancelPendingAppends(handleId: string, reason: string): void {
    const state = pauseStates.get(handleId);
    if (state === undefined) return;
    const queued = state.queue.splice(0);
    for (const item of queued) {
      item.reject(new Error(`session-log: pending append cancelled (${reason})`));
    }
  }
  ```
  Plan 07 main.ts shutdown sequence calls this for every active handle.

### [IMPORTANT] runtime/daemon/file-bus.ts:213–226 — `claimTask` second-step failure on non-ENOENT not handled; throws with orphan claim left behind

When `fsp.rename(pending→claimed)` fails with anything OTHER than ENOENT (EPERM, EACCES, EBUSY on Windows), line 225 re-throws but the `.claim.json` is NOT unlinked. Next call → EEXIST → false-positive `already-claimed` based on a now-defunct claim.

  Why: Windows file-locking (EBUSY) is common when antivirus scans the pending file. The claim is stale immediately but holds the lock until `reclaimIfStale` runs 6 hours later.
  Fix: wrap the rename in try/catch, on ANY error, unlink the claim file before re-throwing (or returning):
  ```ts
  try {
    await fsp.rename(pendingPathOf(taskId), claimedTaskPathOf(taskId));
  } catch (err) {
    await fsp.unlink(claimPath).catch(() => undefined);
    const code = getErrnoCode(err);
    if (code === "ENOENT") {
      return { claimed: false, reason: "already-claimed" };
    }
    throw err;
  }
  ```

### [IMPORTANT] runtime/daemon/file-bus.ts:282–296 — `writeResolvedOutput` tmp file uses dot-prefix `.${taskId}.tmp` — risks collision with `claimTask` if two ops in flight

`resolvedTmpPathOf` (line 119) → `pathFor("tasks/resolved") + "/." + taskId + ".tmp"`. Tmp is in `tasks/resolved/` so no collision with `tasks/claimed/<id>.claim.json`. Two concurrent `writeResolvedOutput` calls for the same taskId from owner-A would both attempt to open `.t.tmp` with mode `"w"` (NOT `"wx"`).   Why: concurrent retries from same owner (or duplicate dispatch from Plan 07 ipc-server) can corrupt tmp. The owner-id check rejects different-owner concurrent writes but NOT same-owner concurrent writes.

  ```ts
  function resolvedTmpPathOf(taskId, attemptId) {

  }
  ```
  This also makes the test at line 158–185 ("cleans up tmp file when atomicRename fails") more robust against future change.

### [IMPORTANT] runtime/daemon/session-log.ts:148–168 — `performAppend` fsync on data only; sequence file write at line 165 has no fsync

`persistSequence` (line 119–121) is plain `fsp.writeFile(..., String(seq))` — no `datasync`. After write resolves, the `.seq` content may sit in OS page cache for seconds. Crash window: log line is on disk (datasync), `.seq` still cached → on restart, recovery reads stale `.seq`, allocates duplicate.

  Why: amplifies C5 above; even without the cache-vs-persist race in C5, the `.seq` write itself lacks durability.
  Fix: durable seq write via `open + writeFile + datasync + close`:
  ```ts
  async function persistSequence(handleId, seq) {
    const seqPath = seqPathOf(handleId);
    const handle = await fsp.open(seqPath, "w");
    try {
      await handle.writeFile(String(seq));
      await handle.datasync();
    } finally {
      await handle.close();
    }
  }
  ```

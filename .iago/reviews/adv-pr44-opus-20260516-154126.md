### Important: 4

[IMPORTANT] runtime/daemon/ipc-server.ts:199-217 — No per-connection buffer cap (memory DoS via never-newlined input)
  Why: The `handleConnection` handler accumulates `buffer += chunk` unbounded until a `\n` arrives. A malicious or buggy client (local same-host, since the socket is now `0o600`, but still: a compromised daemon-user process) can stream arbitrarily many bytes without ever sending `\n`, growing daemon RSS without limit. EC3 chunking test validates the happy path but not the adversarial path. Phase 1 trust model is same-host + owner-only, which lowers severity from Critical to Important — but the daemon should still cap the buffer.
  Fix: Add a per-connection `MAX_LINE_BYTES = 64 * 1024` (or similar) constant. When `buffer.length > MAX_LINE_BYTES` without a newline, write `{ ok: false, error: "parse-error: line-too-long" }` and `socket.destroy()`. Add test 15: 100KB of garbage with no newline → connection closes, daemon RSS unchanged.

[IMPORTANT] runtime/daemon/ipc-server.ts:241-249 — `socketTails` chain extends forever on long-lived connections
  Why: Each `processLine` does `socketTails.set(socket, work)` where `work` is the awaited tail. A connection that sends 10,000 requests builds a 10,000-link Promise chain on that socket's tail. After each handler completes, the previous links are unreachable and GC'd — but only because the WeakMap entry is overwritten each call. The chain itself does NOT cause unbounded growth (each `then` callback runs and returns synchronously, the previous tail's reference is dropped on overwrite), but it does create deep microtask scheduling. On Node 20 this is benign; it becomes a problem only if a handler holds onto `previousTail` externally. Low-severity, but worth a comment + a guardrail.
  Fix: Add JSDoc on `processLine` explaining that the tail chain is bounded by serial drain — once each handler resolves and writes, its tail is no longer referenced. Optional: add an explicit no-op `previousTail.catch(() => {})` to prevent unhandled rejection if `previousTail` somehow rejects (currently the `try` inside the `then` swallows errors from `dispatched` but NOT from `previousTail` itself — a write error on the previous response would bubble as unhandledRejection).

[IMPORTANT] runtime/daemon/ipc-server.ts:198-226 — No per-connection idle timeout
  Why: `handleConnection` does not call `socket.setTimeout(...)`. An idle client connection sits forever consuming a file descriptor + the per-socket state (Set entry, WeakMap entry, accumulator). Same-host owner-only mitigates the threat surface, but the daemon should bound per-connection liveness for hygiene. Especially relevant when Phase 6 dashboard wires multiple browser tabs.
  Fix: After `socket.setEncoding("utf8")` add `socket.setTimeout(5 * 60 * 1000); socket.on("timeout", () => socket.destroy());`. Document the 5-minute idle floor in the Phase 6 forward-list — dashboard long-poll endpoints will need to override.

[IMPORTANT] runtime/daemon/ipc-server.ts:312-328 — `cachedFleetHealthPromise` cached even on rejection
  Why: If `getFleetHealth()` rejects, the `finally` clears `cachedFleetHealthPromise` so the next caller retries — correct. But: the rejected promise reference is what concurrent callers awaited via L315 `return existing`. Those concurrent callers all receive the same rejection — also correct. However, the `cachedFleetHealth` field is NOT updated on rejection, so the cache stays empty → every subsequent request fires a fresh `getFleetHealth()` until one succeeds. If the upstream stays down, this becomes a thundering herd on the failing endpoint (one call per IPC request, no backoff). For Phase 1 hello-world this is acceptable (the injected `getFleetHealth` is in-process), but document.
  Fix: Add a comment at L312 explaining "rejection invalidates cache but does NOT install a cooldown — Phase 6 will add error-state caching to prevent thundering-herd on persistent upstream failure." Optional Phase 1 addition: cache the rejection with a short TTL (e.g., 1s) to absorb bursts; skip if Phase 6 will subsume.

### Minor: 5

  Why: README row 20 reads "IPC server for in-process tool dispatch + Telegram routing" — but Plan 05 scope is `fleet-health` / `list-agents` / `get-handle` dashboard endpoints, NOT Telegram routing (Telegram is Plan 06 file-bus polling, separate channel). The label conflates two transport layers.

  Why: The header documents the 10 event kinds by name but not their payloads — readers must drop into the type definition.   Fix: Optional — extend the header table with `(handleId, alive, rssBytes?)` etc. or leave to the TS type. Acceptable as-is.

[MINOR] runtime/daemon/ipc-server.ts:51-57 — Windows named-pipe path is not parameterized for multi-instance
  Why: Two daemons on the same Windows machine collide on `\\.\pipe\iago-os-v2-daemon`. POSIX path `/tmp/iago-os-v2-daemon.sock` has the same property. Phase 1 is single-instance per box, so this is fine, but Phase 6 dashboard development may spin a second daemon (sandbox + prod). Document.
  Fix: Add JSDoc: "Single-instance assumption — Phase 1 runs one daemon per host. Phase 7 multi-tenant work will need namespaced paths (e.g., `iago-os-v2-daemon-<org>`)."

[MINOR] runtime/daemon/telemetry.ts:88-93 — UTC midnight rollover is documented as M2 but not flagged in any test
  Why: Header L8-10 acknowledges the M2 edge case ("emit beginning before midnight, resolving after, lands in new file — acceptable for Phase 1"), but there is no assertion locking that behavior. If a future refactor changes file-resolution to "capture filename at function-entry time", the M2 contract silently changes.
  Fix: Add test 10 with a `Date.now` mock that advances across midnight between two `emit()` calls; assert both files exist with one line each. Or: explicitly state in the header "no test guards this — change with care."

[MINOR] runtime/daemon/ipc-server.ts:243-244 — Internal-error message leaks underlying exception text to clients
  Why: `internal: ${message}` exposes the raw `err.message` to the IPC client. Trust model is owner-only, so this is informational disclosure to a trusted client — but in a defense-in-depth posture, internal error text should not cross the IPC boundary. Phase 6 dashboard might log or render this text to a UI.
  Fix: Replace with `internal: handler failure` and `console.error("[ipc-server] internal error:", err)` to keep details server-side. Defer if Phase 6 explicitly wants the message for debugging.

### Forward (Phase 6+ scope — NOT a Phase 1 block)

[FORWARD] Named pipe ACL on Windows — Plan 05 ships no explicit ACL.
  Why: Probe 2. Node's `net.createServer().listen("\\\\.\\pipe\\...")` creates a named pipe with default ACL that allows Everyone-readable depending on Windows version/policy. POSIX side is correctly chmod'd 0o600 (L150). Windows side has no equivalent restriction in Node's API surface — would require a native module (e.g., `node-windows-acl`) or PowerShell post-creation `Set-Acl`.
  Disposition: Acceptable for Phase 1 (Santiago's box is single-user). Phase 6 dashboard work must add a Windows ACL step or document the threat. Open as Phase 6 forward-list item.

[FORWARD] IPC authentication — no token, no peer-credentials check.
  Why: Probe 11. Plan 05 trust model is same-host, owner-only (enforced by Unix `0o600`). No application-layer auth. Phase 6 dashboard wired via the IPC layer inherits this trust model; if the dashboard exposes a remote HTTP face onto IPC, the auth gap becomes critical.
  Disposition: Document in `daemon/README.md` "Phase 1 trust model" subsection. Phase 6 dashboard plan MUST address before exposing IPC to a network surface.

[FORWARD] Backpressure on slow IPC consumer — `socket.write` returns boolean, unchecked.
  Why: Probe 13. L246 `socket.write(...)` discards the return value. If the consumer is slow, writes buffer in the kernel + Node TLS buffer unbounded. For Phase 1 hello-world (small responses, fast consumer) this is fine. Phase 6 dashboard with long-running event streams will need `if (!socket.write(...)) socket.once("drain", ...)` flow control.
  Disposition: Forward to Phase 6 streaming work. Not a Phase 1 gap.

[FORWARD] Telemetry event size limit / blob truncation policy — none declared.
  Why: Probe 6. `emit(event, extra)` accepts arbitrary `extra: Record<string, unknown>`. A misbehaving caller passing a 10MB object writes a 10MB NDJSON line — file grows, dashboard NDJSON parsers may choke. No cap, no truncation.
  Disposition: Phase 1 callers (daemon lifecycle code) write small structured events. Phase 6 dashboard streaming and Phase 8 cost-event flow may emit larger blobs. Add a `MAX_EVENT_BYTES` cap when Phase 6 wires event-bus consumption.

## Universal probes — explicit answers

1. **Unix socket file permissions** — PASS. L148-154 explicit `chmod 0o600` after listen; tested (L344-350) on POSIX. Other users blocked.
2. **Windows named-pipe ACL** — FAIL but acceptable for Phase 1 (Forward item above). Same-user single-box trust model.
3. **IPC protocol shape + parser DoS** — Newline-delimited JSON. Parse errors return structured `{ ok: false, error: "parse-error: ..." }`. Buffer-overflow on huge requests: Important #1 above (no cap).
4. **NDJSON write strategy** — Buffered via `fs.promises.appendFile`. NOT per-event fsync. Documented as such. Phase 1 acceptable — telemetry is observational, not durability-critical.
5. **Telemetry file rotation timezone** — UTC explicit (`getUTCFullYear` / `getUTCMonth` / `getUTCDate`). Documented in header L6 ("One file per UTC date"). Midnight race: M2 known edge case, documented as acceptable.
6. **Telemetry event size limits** — None. Forward item.
7. **IPC graceful shutdown** — `stop()` ends sockets, awaits server.close, then awaits `inflight` set with `Promise.allSettled`. Drains correctly.
8. **Stale socket cleanup** — EC1 fix: `doStart` preemptively unlinks before listen, swallows ENOENT, rethrows others. Test L224-239 asserts.
9. **Concurrent dashboard read + daemon write on NDJSON** — Dashboard tail-readers reading partial last lines is standard NDJSON consumer responsibility — split on `\n`, drop trailing non-empty buffer until next read. Telemetry's `appendFile` is atomic for small lines (`fs.appendFile` uses `O_APPEND` on POSIX, which is per-write atomic up to `PIPE_BUF` ≈ 4KB). Acceptable for Phase 1 event sizes.
10. **CLAUDE_CODE_SESSION_ID on ALL events** — PASS. Every `emit()` call resolves the session ID via `resolveSessionId()` and includes it in the line. No per-stage carve-outs.
11. **IPC authentication** — Same-host trust, no token. Forward item.
12. **Cross-platform client code** — `net.createConnection(socketPath)` works the same on Unix sockets + Windows named pipes. Tests use one code path across both platforms (with one skip for `chmod` + one for EC1 stale-file simulation that PowerShell-creates a non-socket file). Symmetric.
13. **Backpressure** — Unchecked. Forward item.

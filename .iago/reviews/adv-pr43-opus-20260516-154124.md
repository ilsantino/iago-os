## Critical

### [CRITICAL] runtime/agent-runtime/pty/claude-pty.ts:382-388 — `restoreFromMarker` discards original env, substitutes daemon's `process.env`
**Why:** `agent-manager.persistAgentConfig` (agent-manager.ts:890-912) intentionally writes only `{agentId, runtimeId, org, cwd, sessionId}` to `pathFor("agents")/<handleId>.json` — env is NOT persisted on disk. The adapter's `restoreFromMarker` reads that file and then synthesizes `env: { ...process.env } as Record<string, string>`. On crash recovery via the standalone adapter path, the restored PTY inherits whatever env the daemon happens to hold at restart time — which differs from the env the agent was originally spawned with (intentional per-agent secrets, working credentials, AWS_PROFILE, ANTHROPIC_API_KEY scoped per agent, etc. all silently disappear or get replaced by daemon defaults).

This is a SHAPE-1-WIDE security and correctness hazard: an agent spawned with a scoped credential set (e.g., a per-client AWS_PROFILE for Sentria) is restored running under whatever the daemon's parent shell had. Could leak Santiago's personal creds INTO a client agent, or strip a required key OUT of it, depending on direction of drift.

Meanwhile, agent-manager.attemptCrashReplay (agent-manager.ts:819-825) does pass `{ ...cfg.env }` from the in-memory `knownConfigs` map — but that path runs the adapter's `restoreFromMarker` first (line 786) and the adapter has already spawned the PTY with the wrong env by then. The agent-manager never gets to override.

**Fix:** Either (a) persist env in `persistAgentConfig` (acceptable only with secret redaction) and have the adapter read it; or (b) make the adapter's `restoreFromMarker` accept a `SpawnOpts` argument so the agent-manager passes the authoritative env; or (c) split: adapter only "validates the marker" / returns null/throws, and the agent-manager spawns via the standard `runtime.spawn(spawnOpts)` path. Option (c) is cleanest — it removes the duplicated spawn logic and matches how `doRestart` (line 303) already works. Also the documented contract for `restoreFromMarker` in the ADR says "re-creates the session handle from the persisted slot record" — but the persisted record TODAY does not carry env. Either the spec or the persistence layer is wrong.

### [CRITICAL] runtime/agent-runtime/pty/claude-pty.ts:380-390 — `restoreFromMarker` ignores HWM contract, returns handle without checking replay viability
**Why:** Plan 04 Task 5 + the adapter's own JSDoc (lines 28-33) + claude-pty.md (lines 122-142) all state: "If the stored HWM is missing, `restoreFromMarker` returns `null` and `agent-manager` falls back to a clean spawn." The implementation NEVER calls `getHWM(handleId)`. It happily returns a fresh handle even when there is no HWM — meaning a session with no recoverable progress gets a new PTY anyway, but `agent-manager.attemptCrashReplay` will then iterate session.jsonl with the HWM the manager fetches later (line 800) and replay events from byte 0. There is no negotiation between adapter and manager about whether replay is even possible.

This silently violates the documented two-phase replay contract. Worse, in the happy path the adapter returns generationToken=1 (line 389) hardcoded — but the agent-manager's standard restart path increments token from `existing.handle.generationToken + 1`. After several restarts, the standalone `restoreFromMarker` path resets the generation counter to 1, breaking the "stale generation token" stale-completion detection documented in the ADR § Shape 2 (also relied on by Shape 1's heartbeat-restart race avoidance).

**Fix:** Either delete the `restoreFromMarker` implementation entirely and let `agent-manager.doRestart` own re-spawn via `runtime.spawn` (preferred — matches the recycle/restart path), OR implement the documented contract: check `getHWM(handleId)`, return null if missing, accept the prior generationToken via marker or signature so the increment is correct.

### [CRITICAL] runtime/agent-runtime/pty/claude-pty.ts:175-183 + 336-370 — `state` is never removed from `stateByHandleId` after graceful exit (SIGTERM path)
**Why:** `shutdown(handle, "SIGTERM")` schedules a 30s escalation timer but DOES NOT delete the state from `stateByHandleId` at any point. The map entry survives. The only paths that remove state are:
- explicit `SIGKILL` shutdown (line 368)
- `_resetPtyAdapterStateForTests` (test-only)

On the normal flow — daemon issues SIGTERM, PTY exits cleanly within grace, `onExit` fires (line 175), `state.alive = false` set — the state is never deleted. It accumulates. Every restart, recycle, or graceful shutdown leaks one handle's worth of: PTY proc object reference, ~4KB outputBuffer, statusListeners set, sigkillTimer reference, env clone (line 220), etc.

Compounded by the SIGKILL escalation: when it fires (line 357), `current.ptyProcess.kill("SIGKILL")` runs but `stateByHandleId.delete` is NOT called from that callback either.

**Fix:** Inside `onExit` handler in `wireDataAndExit` (line 175-183), after setting `state.alive = false`, delete the state from the map: `stateByHandleId.delete(handleId)`. Also clear `state.statusListeners` so the parent agent-manager's listener references release. Add a test asserting the map is empty after `emitExit(0)`.

## Important

### [IMPORTANT] runtime/agent-runtime/pty/prompt-parser.ts:101 — sub-100-byte unrecognized buffer silently classifies as `idle`, fires false status transitions
**Why:** When the parser receives <100 non-whitespace bytes with no pattern match, it returns `{ status: "idle", matchedPattern: null }`. This contradicts the parser's own header comment ("the caller retains the previous status until a real signal arrives") and the test expectation "does not false-positive unknown on short whitespace input" — both suggest "don't change status" but the code DOES return idle.

In the adapter (claude-pty.ts:158), `if (parse.status === state.lastStatus) return;` — but `lastStatus` initializes to `"running"` (line 210). On the very first chunk of Claude's startup banner (almost always sub-100 bytes), the parser returns `idle`, the comparator sees a transition from `running` → `idle`, and listeners fire with `idle` while Claude is actually still initializing. Status telemetry becomes noisy / wrong; the heartbeat (Plan 03) may make decisions on the false `idle` signal.

**Fix:** Return a sentinel like `status: "unknown-below-threshold"` or a new "no-change-signal" mode and update the adapter to skip transitions for that sentinel. Or: thread the previous status into `parseStatusFromOutput` so the function can return the unchanged value explicitly. The current "return idle as default" violates fail-closed posture in the opposite direction.

### [IMPORTANT] runtime/agent-runtime/pty/claude-pty.ts:153-173 — every `onData` re-tests all 4 regexes against the full 4KB; old tool-marker text wedges status at `running`
**Why:** With event-driven parsing (EC2), every chunk triggers `parseStatusFromOutput` over the entire buffer. The idle regex is end-anchored (`$`) but the `running` regex (`/Running tool:|⏵\s|tool_use:|Calling\s\w+\(/i`) matches ANY occurrence in the buffer. After a `running` classification fires (tool execution mid-stream), the buffer keeps growing and eventually Claude returns to `idle`. The transition fires only if the running marker scrolls past the 4KB cap; on quiet sessions where Claude prints an "Running tool: X" line and then waits, the line stays in the 4KB window. Status sticks at `running` even though `Human:` arrives at the tail. Iteration order has `running` evaluated BEFORE `idle` (KNOWN_PATTERNS[2] vs [3]), so the running match wins.

**Fix:** Either (a) make the running pattern end-anchored too — "running" should reflect the CURRENT state, derived from the tail of the buffer, not anywhere in it; or (b) maintain a "since-last-status-change" sub-buffer that resets on each classification. The order-matters precedence (crashed > idle from a single buffer) is fine for safety but breaks for state-machine purposes.

### [IMPORTANT] runtime/agent-runtime/pty/version-pin.ts:43-94 — `getClaudeCodeVersion` spawn timeout path leaks `child.stdout`/`child.stderr` listeners and may hang on Windows
**Why:** When `VERSION_PROBE_TIMEOUT_MS` (5s) fires, the handler calls `child.kill()` and rejects. The `stdoutChunks`/`stderrChunks` arrays are still referenced by the `data` event handlers, which remain attached. If `child.kill()` succeeds, the child eventually emits `close` AFTER `settled` is already true (line 75 early return) — fine for control flow, but the listeners and chunk buffers are never released until the child process emits its terminal event. On Windows under ConPTY where kill semantics differ, the child may not exit promptly. Memory grows per timeout.

Also: `child.kill()` defaults to SIGTERM on POSIX; on Windows node sends a stop signal that may not terminate `claude.exe` cleanly if it's blocked on its own subprocess. The probe can hang the daemon if the wrong process gets stuck.

**Fix:** On timeout, detach the data listeners explicitly: `child.stdout.removeAllListeners("data"); child.stderr.removeAllListeners("data")`. Consider SIGKILL fallback after the kill timeout. Also: add `child.unref()` so the probe child can't keep the event loop alive on graceful shutdown.

### [IMPORTANT] runtime/agent-runtime/pty/claude-pty.ts:198-204 — PTY spawn does NOT propagate `CLAUDE_CODE_SESSION_ID`; session-log keyed on `handleId` not `sessionId`
**Why:** The user's review brief explicitly probed this: "CLAUDE_CODE_SESSION_ID: where is it generated? Per L2 acceptance criterion #5, all telemetry events must be keyed on this — does the PTY adapter emit telemetry with that key?"

Inspection: the adapter generates `handleId = crypto.randomUUID()` (line 197) and the `AgentHandle` carries `sessionId: opts.sessionId` (passed in from `SpawnOpts`). NOTHING propagates `CLAUDE_CODE_SESSION_ID` into the spawned PTY's env. The `claude` binary will pick its own session id on each spawn, so on restart/replay the PTY sees a brand new Claude-internal session id. Cross-correlation between iaGO's session.jsonl events and Claude Code's own session log is impossible.

Also: `appendEvent(handleId, ...)` (line 138, and session-log.ts:75 `logPathOf` = `pathFor("session-logs")/<handleId>.jsonl`) is keyed on `handleId`, NOT `sessionId`. Restart bumps `handleId` (line 197 always randomUUIDs). After a recycle, the session log is split across two files — `<handleId-1>.jsonl` and `<handleId-2>.jsonl` — even though they belong to the same logical sessionId. The two-phase replay loop in `agent-manager.attemptCrashReplay` (line 802 → readEventsUpToHWM with the OLD handleId) reads from the prior file, but the resumed agent gets a fresh handleId so subsequent events land in a NEW file. The session log is fragmented across incarnations.

**Fix:** (1) Set `env: { ...opts.env, CLAUDE_CODE_SESSION_ID: opts.sessionId }` in the PTY spawn opts. (2) Decide whether session-log should key on `sessionId` or `handleId` — pick one and audit all callers; my read is `sessionId` is correct for the file naming (one log per logical session), with `handleId` only as an event-tag inside the entries. (3) Add a test that two consecutive spawns with the same sessionId append to the same session.jsonl.

### [IMPORTANT] runtime/agent-runtime/pty/claude-pty.ts:290-315 — `send({kind:"prompt"})` and `send({kind:"inject"})` do NOT persist input events; replay loop has nothing to feed
**Why:** Per the claude-pty.md replay flow (lines 122-138): "Only `prompt` and `inject` events are re-sent" during `restoreFromMarker`. The agent-manager's `eventToReplayableMessage` (visible from grep — agent-manager.ts:880-884) builds an `inject` message reading `event.payload.text` from session.jsonl. But the adapter's `send` only calls `appendEvent` for STATUS transitions (line 138 inside `persistStatusEvent`). The actual prompt/inject TEXT the daemon sent is never logged.

Result: replay reads only status events, `eventToReplayableMessage` returns null for every one of them, and the replay loop is a complete no-op. The whole crash-recovery story (the documented Plan 04 deliverable) does not work even when HWM is present — because there is nothing in the log to replay.

**Fix:** In `send` (line 290-315), AFTER successful `ptyProcess.write`, persist the message via `appendEvent(handleId, { kind: message.kind, payload: message.payload, at: Date.now() })` for `prompt` + `inject` (NOT for approval/abort/custom). Add a test that calls `send({kind:"prompt", ...})` and asserts `appendEvent` was invoked with the right shape. Replay will not function without this.

### [IMPORTANT] runtime/agent-runtime/pty/claude-pty.ts:392-395 — `inject` method uses `this.send` and breaks under destructuring
**Why:** `inject(handle, text)` does `await this.send(handle, ...)`. The adapter object is exported as `claudePty: PTYAdapter` — a plain object literal. If a caller does:

```ts
const { inject } = claudePty;
inject(handle, "x");  // TypeError: cannot read 'send' of undefined
```

Or after passing the method around as a callback (likely in the plan-06 Telegram `/inject` router). In test code, callers DO invoke `adapter.claudePty.inject(handle, ...)` so `this` is bound — but production wrappers that store a method reference will explode.

**Fix:** Inline the dispatch in `inject` (call `stateByHandleId.get(handle.id)` + write directly, no `this`), OR bind explicitly at definition time. Add a test that destructures `inject` from the adapter and confirms it still works.

### [IMPORTANT] runtime/agent-runtime/pty/claude-pty.ts:198 — no validation on `opts.cwd` or env keys; arbitrary cwd / env-injection vectors
**Why:** `ptySpawn("claude", [], { cwd: opts.cwd, env: { ...opts.env }, ... })`. `node-pty` on Windows uses ConPTY which forks via CreateProcess; env values containing CR/LF or null bytes can break the env block. On POSIX, control bytes in env values are passed through. `cwd` is passed to fork's chdir — a path with embedded null byte throws, but a path with parent-directory escape (`../../etc`) is allowed by node-pty without validation.

The adapter has no validation. Plan 06 wires Telegram → agent input; if any callsite eventually constructs `SpawnOpts` from a string the daemon doesn't fully trust (e.g., a per-agent config Santiago hand-edits, or a Telegram-controlled agent-launch flow), this becomes an arbitrary-cwd-escape vector. The state-paths module already exposes `assertSafeIdentifier` (used by markers) — no equivalent for cwd/env in the adapter.

**Fix:** Document in claude-pty.md "Known limitations" that callers MUST validate cwd is within the configured agent workspace root, and env keys match `^[A-Z_][A-Z0-9_]*$`. Defense-in-depth: add a check inside `spawnInternal` for path traversal (resolve cwd, check it's under a configured `IAGO_AGENT_WORKSPACE_ROOT`).

## Minor

### [MINOR] runtime/agent-runtime/pty/claude-pty.ts:89 — `MAX_BUFFER_BYTES = 4 * 1024` is computed in BYTES but truncation operates on STRING length (UTF-16 code units)
**Why:** `truncateBuffer` (line 104-110) does `joined.length` and `joined.slice(joined.length - MAX_BUFFER_BYTES)`. JavaScript `string.length` is UTF-16 code units, not bytes. A buffer of 4096 BMP chars = 4096 code units = up to ~12288 UTF-8 bytes. Probably fine for Claude's mostly-ASCII output, but the variable name is misleading.

**Fix:** Rename to `MAX_BUFFER_CHARS`, OR use `Buffer.byteLength(joined, "utf8")` for the truncation decision.

## Forward (defer-explicit, document in plan)

### [FORWARD] PTY shutdown does NOT use process-group / job-object containment — orphans on parent crash
**Why:** Probe 10. node-pty fork on POSIX puts the child in its own session (setsid). When the daemon SIGKILLs itself or crashes hard, claude.exe and any subprocesses it spawned continue to run. On Windows, node-pty's ConPTY does not enroll the child in a job object that the daemon owns.

**Action:** Document as known limitation in claude-pty.md "Known limitations". Future plan (Phase 1 cleanup or Phase 7 hardening) wraps node-pty spawn in a Job Object on Windows + cgroup on Linux for hard-containment.

### [FORWARD] Concurrent `send({kind:"inject"})` calls are NOT serialized — interleaving hazard
**Why:** Probe 12. Two simultaneous calls to `state.ptyProcess.write(textA + "\n")` and `state.ptyProcess.write(textB + "\n")` are not coordinated. If the PTY is mid-render (Claude printing tool output), the writes can interleave into a single line Claude parses as one prompt: `textAtextB`. No mutex.

**Action:** Document in claude-pty.md. Future plan adds a per-handle async mutex around `send`.

### [FORWARD] No token redaction policy yet — safe only because env never lands on disk
**Why:** Probe 11. The adapter prints env nowhere directly, and `persistAgentConfig` does NOT write env to disk. Safe by accident. If a future change adds env persistence (suggested in CRITICAL #1 fix), redaction MUST be added at the same time.

**Action:** When implementing env persistence (CRITICAL #1 fix), add a redactor stripping keys matching `/_TOKEN|_KEY|_SECRET|PASSWORD|API_KEY/i`. Test it. Document in the persistAgentConfig JSDoc.

### [FORWARD] No PTY resize support; cols/rows hardcoded 200×50
**Why:** Probe 13. `PTY_COLS = 200; PTY_ROWS = 50` (lines 91-92) hardcoded. Claude Code adapts output to terminal width. A real terminal user resizing the dashboard does not propagate to the PTY.

**Action:** Document in claude-pty.md "Known limitations". Phase 6 dashboard plan owns resize wiring.

### [FORWARD] No `$CLAUDE_BINARY` override; bare-name PATH lookup only
**Why:** Probe 4. The adapter calls `ptySpawn("claude", [], ...)` with the binary as a bare name. The version probe similarly uses `spawn("claude", ...)`. No escape hatch for multiple installs, beta builds at specific paths, or pre-release testing.

**Action:** Phase 1 cleanup: honor `process.env.CLAUDE_BINARY` if set, else fall back to `"claude"`. One-line change. Not blocking for Plan 04 ship.

## Summary

- **Plan compliance:** FAIL (4 gaps — restoreFromMarker HWM, sessionId propagation, input event persistence, capture.ps1)
- **Garry:** FAIL (5 threads)
- **Critical:** 3
- **Important:** 9
- **Minor:** 6
- **Forward:** 5

The PR ships a working adapter for the happy-path single-spawn case but the crash-recovery semantics (the whole reason the AgentRuntime contract has `restoreFromMarker`) are broken in three independent ways: env not propagated, HWM not checked, generation token reset. Replay is functionally a no-op because input events are not persisted. Session-id correlation with Claude Code's internal session log is impossible without env propagation. Three of these would have been caught by the pipeline review stage that was skipped.

Recommend: BLOCK on the 3 Critical findings (env restoration, HWM contract, state-map leak). Important findings — especially session-id propagation and input-event persistence — should be addressed before any reliance on crash recovery in Plan 07's hello-world acceptance gate. Garry threads (tmp artifacts, capture.ps1) should be resolved before merge regardless.

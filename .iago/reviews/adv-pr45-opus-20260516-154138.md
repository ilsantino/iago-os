## CRITICAL

### [CRITICAL] runtime/telegram/bot.ts:118-143 — Allowlist NOT applied to `chat.id`; group-chat hijack possible
**Why:** `handleMessage` checks `from.id` against `allowedUserIds` but the reply is sent to `msg.chat.id`. If Santiago is in any group chat with the bot added, a Santiago message in that group authenticates as Santiago's `from.id`, and the bot's reply (containing agent state, handle IDs, pending approvals, command output, even token-adjacent error strings from `injectIntoAgent`) is sent to `msg.chat.id` — the group chat. A `/agents` or `/status` reply leaks internal agent topology to every group member. There is no `chatId === expectedChatId` check anywhere in `handleMessage` / `handleCallbackQuery`.
**Fix:** add a chat-allowlist check. Simplest: gate on `msg.chat.type === "private"`. Reject silently (stderr log) on group/supergroup/channel chats. Apply identically in `handleCallbackQuery`.

```ts
if (msg.chat.type !== "private") {
  console.error(`[telegram] rejected non-private chat (${msg.chat.type}, id=${msg.chat.id})`);
  return;
}
```

### [CRITICAL] runtime/telegram/bot.ts:315-345 — `/inject` accepts arbitrary text to PTY stdin with zero control-byte sanitization; no length cap
**Why:** `dispatchInject` validates the `agentId` via `validateAgentId()` but the `text` argument is passed verbatim to `injectIntoAgent` which (per Plan 04 `claude-pty.ts`) writes raw bytes to the PTY's stdin. Any well-formed `agentId` followed by ANY text — including `\n`, `\r`, `\x03` (Ctrl-C), `\x04` (EOF), `\x1b]52;c;...\x07` (OSC 52 clipboard injection), or megabyte-scale dumps — is forwarded. Telegram's per-message cap bounds single-message size but does not bound a sequence of `/inject` calls. `\n` in the text gives the user (or anyone who phishes Santiago's Telegram session) shell-like ability to submit any sequence of prompts or terminal-control sequences to mess with the running TUI.
**Fix:** at minimum (a) reject text containing control bytes outside `\t \n` via regex `/[\x00-\x08\x0b\x0c\x0e-\x1f\x7f]/`, (b) cap injected text length (e.g., 2048 bytes per message). If injection of terminal-control sequences IS intended, document that explicitly in `runtime/telegram/README.md` § Security model.

### [CRITICAL] runtime/telegram/bot.ts:84-91 — Empty `allowedUserIds` accepted silently; `chatId` becomes `undefined`; approvals silently fail
**Why:** Constructor accepts `allowedUserIds: number[]` with no length validation. If misconfigured to `[]` (e.g., `IAGO_TELEGRAM_ALLOWED_USER_IDS` unset → `config.ts:328` returns `fileTelegram?.allowedUserIds ?? []`), then `this.chatId = opts.allowedUserIds[0]` is `undefined`, and `this.allowedUserIds = new Set([])`. Two consequences: (a) NO user can send commands (bot is dead, silently); (b) `sendApprovalRequest` posts to whatever `chatId` the caller supplies — but the daemon's bot-side poller (Plan 07) calls `bot.getChatId()` first, which returns `undefined`. Result: approvals silently fail to broadcast, agents block on `waitForApproval` until timeout. No telemetry event, no stderr log. This is a hard-pause failure mode in production.
**Fix:** constructor validates `allowedUserIds.length >= 1` and throws `RangeError("TelegramBot: allowedUserIds must be non-empty")`. Fail loud at startup.

## IMPORTANT

### [IMPORTANT] runtime/telegram/bot.ts:122 — Allowlist accepts `id <= 0`; `from === undefined` untested
**Why:** `Set<number>.has(from.id)` is strict; type-coercion risk is low. But `parseAllowedUserIds` in `config.ts:95-103` happily accepts `0` and negative integers; if an operator typos `0` into the config, that user ID (which never occurs in real Telegram but could in spoofed updates from a compromised proxy) matches. The `from === undefined → return` guard at line 121 is correct but has no test asserting silent drop.
**Fix:** in `parseAllowedUserIds` reject `id <= 0` (Telegram user IDs are always positive). Add a test asserting `from === undefined` is silently dropped without crashing.

### [IMPORTANT] runtime/telegram/approval-bus.ts:204-262 — `resolveApproval` race-loser window can return `not-found` for a winning approval
**Why:** The atomic claim point is `fs.unlink(pendingPath)`. The winner: (1) unlink succeeds, (2) writes resolved file. A concurrent loser between step 1 and step 2 sees `ENOENT` on unlink, then polls for the resolved file with 5 retries × 20ms = 100ms total. If the winner's write takes >100ms (slow disk, antivirus inspection on Windows, NFS lag), the loser returns `{ ok: false, reason: "not-found" }` — semantically wrong. The bot then replies `"Approval <id>: not-found"` to the user despite the winner's resolve eventually succeeding.

Tests do not exercise slow-disk conditions; the 10-way concurrent test (`approval-bus.test.ts:198-220`) passes because in-memory tmp-dir writes complete sub-millisecond. README (line 60-63) advertises "first-tap-wins" but the implementation can lie about who lost.
**Fix:** (a) bound the resolved-write phase with a generous timeout (e.g., 5s of polling) before classifying as `not-found`, OR (b) flip the order — write the resolved-tmp file FIRST, then atomic-rename-replace pending into resolved/, eliminating the gap.

### [IMPORTANT] runtime/telegram/bot.ts:97-103,105-116 — `stop()` does not call `removeAllListeners`; failed `stopPolling` leaves zombie polling client
**Why:** `stop()` sets `this.bot = null` (line 115) without `removeAllListeners()`. If `stopPolling` throws (try/catch line 108-114), the old `TelegramBotApi` instance remains polling Telegram and consuming the update offset. Next `start()` creates a SECOND polling client → Telegram returns HTTP 409 on next `getUpdates`. Test `bot.test.ts:377-383` asserts second `start()` doesn't crash but does NOT verify single-instance polling.
**Fix:** in `stop()`, call `this.bot.removeAllListeners()` before nulling. Log loudly if `stopPolling` fails. Add a test: start → stop (with stopPolling throwing) → start → assert single factory invocation.

### [IMPORTANT] runtime/telegram/bot.ts:97-99,100-102 — Message/callback handlers fire-and-forget; unhandled rejections silently dropped
**Why:** `this.bot.on("message", (msg) => { void this.handleMessage(msg); })` discards rejected promises. `handleMessage` try/catches known paths, but an exception in `parseCommand` (e.g., exotic Unicode in `text.trim().split(/\s+/)`) bubbles to `void` and is silently dropped. No `process.on("unhandledRejection")` installed.
**Fix:** wrap `handleMessage` body in a top-level try/catch that logs to stderr AND emits a `telegram-handler-error` telemetry event.

### [IMPORTANT] runtime/telegram/bot.ts:145-182 — Callback queries never call `answerCallbackQuery` on rejection paths
**Why:** Non-allowed user fires callback (line 150) → return without `answerCallbackQuery`. Telegram shows the loading spinner until it times out (~15s). Parse failures (line 167-170) also skip `answerCallbackQuery`. Only the success path calls it. UX: tapping an inline button as a non-allowed user shows a stuck spinner — confusing AND an info leak (bot reacts to events from this user even though no message is sent).
**Fix:** call `bot.answerCallbackQuery(query.id, { text: "Not authorized" })` for non-allowed; `answerCallbackQuery(query.id, { text: "Invalid callback" })` for parse failures. Move into a `finally`.

### [IMPORTANT] runtime/.gitignore + README:128-147 — No `.env.example`; runtime/.gitignore omits `.env` patterns; no token rotation procedure documented
**Why:** Repo root `.gitignore` covers `.env`/`.env.*`, but Plan 06 ships `runtime/.gitignore` (9 lines) that does NOT include `.env` patterns. Defense-in-depth fails open if a developer creates `runtime/.env` under the assumption the local `.gitignore` covers it. No `.env.example` is shipped. README does not document token rotation if the token leaks.
**Fix:** (a) add `.env`, `.env.*`, `!.env.example` to `runtime/.gitignore`; (b) ship `runtime/.env.example` with `IAGO_TELEGRAM_BOT_TOKEN=`, `IAGO_TELEGRAM_ALLOWED_USER_IDS=`, `IAGO_TELEGRAM_CHAT_ID=` placeholders; (c) document rotation: BotFather `/revoke`, update env, restart daemon.

### [IMPORTANT] runtime/telegram/bot.ts:62-91 — Bot token retained as plain `this.token`; no inspect-redactor; leaks via `util.inspect` / serialization
**Why:** Constructor stashes the token on `this.token`. Any error that serializes the bot instance (`JSON.stringify`, `util.inspect`, console.dir on debug paths) leaks the token. No `[Symbol.for("nodejs.util.inspect.custom")]` redaction. No comment marking the field as secret.
**Fix:** define an inspect-redactor:
```ts
[Symbol.for("nodejs.util.inspect.custom")]() {
  return { token: "[REDACTED]", started: this.started };
}
```
Add a test that `util.inspect(bot)` does not contain the token string.

### [IMPORTANT] runtime/telegram/bot.ts:118-127 — Non-allowed user rejection logs `from.username` and `from.id` to stderr (PII)
**Why:** `[telegram] rejected message from non-allowed user ${from.id} (@${from.username ?? "?"})` writes rejected user's Telegram ID + username to stderr. On VPS this lands in journald, persists per retention, and is doxx-grade PII (username → real person). Phase 1 is local-only, but README directs Phase 2 → VPS without changing this logging.
**Fix:** log only a per-window count of rejections, or log a hash of the user ID. Document the PII implication in README § Security model.

### [IMPORTANT] runtime/telegram/bot.ts:384-419 — `sendApprovalRequest` does not validate `chatId` against `allowedUserIds`
**Why:** Signature accepts ANY `chatId` from caller. If main.ts or a future feature plumbs a user-influenced chatId, an attacker who can influence the chatId redirects approval requests to their own Telegram chat and receives `Allow` callbacks.
**Fix:** validate `chatId` is in `allowedUserIds` (or equal to `this.chatId`) before sending. Reject with stderr log otherwise. Add a test.

### [IMPORTANT] runtime/telegram/commands.ts + approval-bus.ts — `approvalId` from user input never validated via `assertSafeIdentifier`
**Why:** `/approve_allow_<id>` parses by slicing; `<id>` then any string. `assertSafeIdentifier` is NOT called on `approvalId` anywhere in the resolve path. Hand-crafted `/approve abc/../../etc/passwd allow` resolves to filesystem path `approvals/pending/abc/../../etc/passwd.json`. Most likely returns ENOENT (no exploit), but on a misconfigured machine could traverse out. Defense-in-depth missing at trust boundary.
**Fix:** call `assertSafeIdentifier(approvalId, "approvalId")` in `resolveApproval`, `waitForApproval`, and `parseCommand` after slicing the callback id.

### [IMPORTANT] runtime/telegram/approval-bus.ts:334-364 — `listPendingApprovals` swallows ALL parse failures silently
**Why:** Malformed JSON in `approvals/pending/` (e.g., half-written file from a crashed `createApprovalRequest`, or a file dropped by a different process) is silently skipped. No stderr log, no telemetry event. Santiago's `/status` shows fewer pending approvals than actually exist on disk; broken file lingers forever. Combined with Plan 07's dedupe marker, this is an invisible stuck-approval class.
**Fix:** on parse failure, log to stderr with offending filename, emit `kind: "approval-malformed"` telemetry event. Optionally rename to `.broken` to surface the issue.

### [IMPORTANT] runtime/telegram/bot.ts + README — Multi-bot-instance hazard (Telegram 409) is NOT documented or handled
**Why:** Stress note in Plan 06 silent on Telegram's "one polling client per token" constraint. If Santiago runs the daemon twice, or if OpenClaw still polls the same token during Phase 2 cutover, Telegram returns HTTP 409 on `getUpdates` and node-telegram-bot-api emits `polling_error`. Bot has no `bot.on("polling_error", ...)` handler. Santiago sees "the bot is not responding" with no signal.
**Fix:** in `start()` register `this.bot.on("polling_error", ...)` to log + emit telemetry. Document in README § Failure modes.

### [IMPORTANT] runtime/telegram/bot.ts:234-251, 347-377 — `/agents` and `/status` replies have no Telegram 4096-char cap protection
**Why:** With 100+ agents (future state), reply exceeds 4096 chars → Telegram 400. `safeReply` catches and logs but user sees no reply. Same hazard in `/status` if `listPendingApprovals` returns many entries.
**Fix:** chunk replies at 4000 chars (headroom for UTF-8 surrogate pairs). Send multiple messages. Document the cap.

## MINOR

### [MINOR] runtime/telegram/approval-bus.ts:305-319 — `waitForApproval` has no AbortSignal hook
**Why:** Caller cannot cancel a long `waitForApproval` (e.g., on agent shutdown). If `timeoutMs` is high (1 hour) and agent is shut down, loop runs uselessly until deadline.
**Fix:** accept optional `AbortSignal`; resolve with `{ aborted: true }` on signal.

### [MINOR] runtime/telegram/commands.ts:54-56 + bot.ts /inject — `tokenize` collapses ALL whitespace; `/inject` payload loses formatting
**Why:** `text.trim().split(/\s+/)` then `tokens.slice(2).join(" ")` collapses whitespace runs, drops newlines. Santiago cannot inject multi-line content.
**Fix:** for `/inject`, parse agent name then use original `text.slice(prefix.length)` to preserve formatting.

### [MINOR] runtime/telegram/README.md:174-181 — library choice "battle-tested" lacks audit evidence
**Why:** `node-telegram-bot-api@^0.65.0` pinned but no `npm audit` evidence in README. The library has had historical CVEs (token-leak in error messages).
**Fix:** run `npm audit --omit=dev --json | grep telegram` once, paste result + date into README. Re-verify in Phase 5.

### [MINOR] runtime/telegram/bot.test.ts:146-150 — `flushTicks(5)` is heuristic
**Why:** Tests rely on 5 `setImmediate` flushes to drain promises. If dispatch chain gains an awaited tick, tests silently pass with no assertion. `waitForFile` is the right pattern.
**Fix:** replace `flushTicks` with explicit `await waitForFile` or `expect(...).resolves` where possible.

### [MINOR] runtime/telegram/bot.ts:319-326 — `validateAgentId` error reply embeds invalid agent id; minor reflected-content surface
**Why:** Reply embeds `command.agent` verbatim. Telegram renders plaintext (no XSS), but an attacker who somehow bypasses allowlist could trigger replies containing arbitrary text → phishing-in-chat-history.
**Fix:** truncate `command.agent` to 64 chars in reply.

### [MINOR] runtime/telegram/bot.ts:347-377 — `/status` reply omits `lastStatus`/liveness probe
**Why:** Plan 06 Task 5 (line 62) requires "alive state + last status + pending approvals". Implementation reports `generationToken` and does not call `agentManager.isAlive` or equivalent.
**Fix:** extend `AgentManagerInterface` with `getLastStatus(handleId)` / `isAlive(handleId)`, include in reply.

### [MINOR] runtime/telegram/commands.ts:69 — Leading-slash normalization for callbacks is fragile
**Why:** Normalization works but is non-obvious; bot's `sendApprovalRequest` emits callback_data without leading `/`.
**Fix:** stylistic — strip leading slash everywhere and use plain prefixes, OR document the asymmetry inline.

### [MINOR] runtime/telegram/bot.ts findHandleByAgentId — `dispatchAbort` / `dispatchStatus` do NOT call `validateAgentId` before lookup
**Why:** Hostile agentId (containing `__`) silently returns null; bot replies "no handle". Acceptable but clearer error from upfront validation.
**Fix:** call `validateAgentId` at the top of `dispatchAbort` / `dispatchStatus` for consistency with `dispatchInject`.

### [MINOR] runtime/telegram/approval-bus.ts:106-109 — nonce helper could be DRYer
**Why:** `pendingTmpPath` / `resolvedTmpPath` duplicate nonce generation. Trivial.
**Fix:** extract a `nonce()` helper.

## FORWARD (Phase 2+ / out of Phase 1 scope, record now)

### [FORWARD] Webhook vs polling — Phase 2 VPS cutover migration path not documented
README doesn't address Phase 2 webhook migration. Production HTTPS-fronted webhook scales better and avoids 409-conflict hazards.

### [FORWARD] `getUpdates` offset persistence on daemon crash
node-telegram-bot-api persists offset internally per-process. After a crash, on next start the bot may re-process the last batch (Telegram retains updates for 24h). Document acceptable replay semantics.

### [FORWARD] `appr_*` legacy passthrough operator runbook
Plan 06 stress PR3 DROPS `appr_*` form; README documents the rename. An ops-runbook entry should warn operators reading old cortextOS docs.

### [FORWARD] Persistent approval audit trail
Only `resolved/<id>.json` files + `approval-resolved` telemetry event. For compliance / post-mortem, a dedicated `telemetry/approvals-<date>.ndjson` stream would help. Document retention story.

### [FORWARD] `/send` command landing in Phase 3
Master prompt § 274-281 specifies `/send <agent> <message>` for Shapes 2-5. Gating logic in place. Confirm Phase 3 plan picks this up + implements payload-schema validation per ADR `custom` rule.

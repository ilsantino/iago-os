## Summary
The plan stack is directionally coherent, but it is not yet safe to hand to implementers as an execution contract. Two issues are blockers: the documented script paths drift from the spec/runbook paths, and the rollback `jq`-over-SSH token patch is fragile enough to break the emergency path during production cutover. Most other findings are important plan-hardening gaps: GH token provisioning is assigned to the wrong wave, archive encryption lacks a real decrypt round-trip, idempotency/race claims are under-specified, and Plan 04’s cron scheduler task has expanded beyond “minimal PR triage.” The inline stress-test sections already notice several of these, but many fixes are only “forward-list” notes, not incorporated into the actual tasks.

## P0 — Blockers
### P0-1: Rollback token patch uses unsafe nested SSH/jq quoting and can break the emergency path
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/03-cutover-rollback-orchestration.md:41`
- **Evidence:** Plan 03 instructs: `tailscale ssh root@$VPS_HOST -- "su - ilsantino -c \"... jq --arg t \\\"$FRESH_TOKEN\\\" '.channels.telegram.botToken = \\$t' ...\""` while the spec’s canonical rollback snippet is multi-line inside remote `su - ilsantino -c` at `.iago/research/2026-05-16-phase-2-vps-bootstrap-spec.md:1277-1284`. Plan 03’s own stress test flags this as quote-hell and recommends a temp script at lines 98 and 125.
- **Risk:** Rollback is the safety net for a production FAST cutover. A quoting error, local `$t` expansion, or token exposure via command argv can prevent OpenClaw restart when Telegram is already broken.
- **Action:** Replace the inline nested command in the task itself with the temp-script-over-SSH/scp pattern from the stress test. Pass the fresh token via stdin or a protected remote env file, never as an interpolated command-line argument. Add a rollback test that patches a fixture JSON and asserts the exact output.

### P0-2: Spec/runbook script paths drift, so the operator and automation can run different files
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/03-cutover-rollback-orchestration.md:34`
- **Evidence:** CONTEXT requires every command/path to land verbatim or with reasoning at `.iago/plans/feature-phase-2-vps-bootstrap/CONTEXT.md:28`. The spec cutover command runs `/opt/iago-os/scripts/vps-bootstrap/archive-openclaw.sh` at `.iago/research/2026-05-16-phase-2-vps-bootstrap-spec.md:1095-1096`, but Plan 02 creates `runtime/deploy/archive-openclaw.sh` at `.iago/plans/feature-phase-2-vps-bootstrap/02-openclaw-teardown-scripts.md:21`, and Plan 03’s executable invokes `/opt/iago-os/runtime/deploy/archive-openclaw.sh` at line 34. Plan 03 also says the human runbook copies spec §8 verbatim at line 48.
- **Risk:** Santiago following the human runbook can execute a non-existent path while `cutover.sh` uses a different path. That breaks T+00 archive/stop and undermines the “verbatim command” guarantee.
- **Action:** Pick one canonical deploy path and update spec-derived plan tasks, runbooks, and scripts consistently. If `runtime/deploy` intentionally replaces `scripts/vps-bootstrap`, document the explicit path translation once and make every copied command use the translated path.

## P1 — Important
### P1-1: `gh-token` provisioning is assigned to Plan 04 after Plan 01 owns the credential map
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/04-pr-triage-agent.md:70`
- **Evidence:** Migration scope requires `GH_TOKEN: ${CRED:gh-token}` at `.iago/research/2026-05-16-v2-operational-migration-scope.md:270`. Plan 01’s CRED_MAP only lists Telegram plus three Anthropic credentials at `.iago/plans/feature-phase-2-vps-bootstrap/01-daemon-deploy-infrastructure.md:43`. Plan 04 later edits Plan 01 artifacts at line 70, and the stress test acknowledges the wave conflict at lines 116 and 136.
- **Risk:** If Plan 01 is implemented or reviewed independently, the deploy script and systemd unit can ship without the credential required by the locked first-real-workflow.
- **Action:** Move `gh-token` into Plan 01 Task 2 and the systemd unit/cred-bootstrap tasks directly. Plan 04 should verify the entry exists, not edit upstream plan-owned artifacts.

### P1-2: Age archive integrity is not verified with a real decrypt round-trip
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/02-openclaw-teardown-scripts.md:41`
- **Evidence:** Spec shows decrypt verification using Santiago’s private key at `.iago/research/2026-05-16-phase-2-vps-bootstrap-spec.md:830-833`. Plan 02 tests stub `age` and only assert sentinel output at line 41. Its stress test admits no decrypt verification at line 106 and proposes only an age header check.
- **Risk:** A wrong/corrupt recipient pubkey can produce an archive that looks successful but cannot be decrypted after the 30-day retention window starts.
- **Action:** Add a test mode using a generated ephemeral age keypair to prove tar → age encrypt → age decrypt → tar list succeeds. Keep the production VPS path public-key-only, but require a real local round-trip in CI/test evidence.

### P1-3: Archive script claims idempotency while re-runs create additional credential archives
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/02-openclaw-teardown-scripts.md:34`
- **Evidence:** Task 1 says the script has idempotency rationale at line 34. Task 2 defines the “idempotent re-run” expectation as a new tarball with a different timestamp and a second manifest row at line 41.
- **Risk:** Re-running the script is not same-state idempotent. It increases the number of encrypted credential snapshots, complicates retention and manifest audit, and can hide accidental double-cutover attempts.
- **Action:** Define idempotency precisely. Prefer a lock plus a sentinel/current archive record: default re-run should detect an existing archive for the same cutover and exit 0 without creating a duplicate, with `--force-new-archive` for explicit extra snapshots.

### P1-4: Credential telemetry contract is inconsistent and lacks a negative leak test in the main tasks
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/01-daemon-deploy-infrastructure.md:78`
- **Evidence:** The spec requires `cred-bootstrap-loaded` with `credentialsLoaded: ["iago-telegram-token"]` at `.iago/research/2026-05-16-phase-2-vps-bootstrap-spec.md:1404-1407`. Plan 01 Task 7 says to emit “names of env vars that were set” at line 78. The no-value assertion only appears in the stress-test section at line 118 and forward-list line 143.
- **Risk:** Implementers can record env var names instead of credential names, and no required task assertion prevents accidentally logging token values to telemetry, console, or journald.
- **Action:** Make the task contract explicit: telemetry records credential file names only, never env var values or secret bytes. Add a required unit test that seeds a fake token and asserts telemetry/journal buffers do not contain the literal value.

### P1-5: Hypothesis: concurrent cutover/archive runs can race because no global lock is specified
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/03-cutover-rollback-orchestration.md:34`
- **Evidence:** Plan 03 adds confirmation and resume flags at line 34 but no cutover-wide lock. Plan 02’s stress test identifies non-atomic manifest append and recommends `flock` at `.iago/plans/feature-phase-2-vps-bootstrap/02-openclaw-teardown-scripts.md:112`. The retention service deletes archive files with `find ... -mtime +30 -delete` per spec at `.iago/research/2026-05-16-phase-2-vps-bootstrap-spec.md:772`.
- **Risk:** Two terminals can run cutover/archive/rollback concurrently, corrupt manifest rows, double-rotate tokens, or interleave NDJSON evidence. Retention deletion can also leave stale manifest rows unless pruning updates the manifest or logs deleted files.
- **Action:** Add a single VPS-side lock for cutover/rollback/archive, plus a separate manifest `flock`. Make the retention service log deleted files and either prune manifest rows or mark them expired.

### P1-6: Plan 04 Task 7 is a scope blowup beyond minimal PR triage
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/04-pr-triage-agent.md:77`

### P1-7: Migration-scope OpenClaw cron inventory is not represented in the plan stack

### P1-8: Cron concurrency control from migration scope is dropped
- **File:** `.iago/plans/feature-phase-2-vps-bootstrap/04-pr-triage-agent.md:42`
- **Evidence:** Migration scope’s cron schema includes `maxConcurrent: 1` at `.iago/research/2026-05-16-v2-operational-migration-scope.md:533-537`. Plan 04’s `crons.json` omits it at line 42, and the scheduler API in Task 7 has no `maxConcurrent` field at line 77.

- **Action:** Carry `maxConcurrent: 1` into `crons.json`, scheduler registration, telemetry, and tests. Add a fake-timer test where one run is still active when the next tick fires.

### P1 (Important)

- **runtime/agent-runtime/pty/claude-pty.ts:398-406 — Crash recovery drops the original agent environment** [high]
  `restoreFromMarker` rebuilds `SpawnOpts` from the persisted config but substitutes the daemon's ambient `process.env` instead of the `env` originally used to spawn the agent. `RegisterAgentConfig` and `SpawnOpts` already model per-agent env, so after a crash replay a recovered PTY can lose API keys, feature flags, or tenant-specific credentials, or worse run under daemon-global credentials while the manager tracks it as the original agent.
  - **Action:** Do not restore a PTY unless the original env is available. Persist the required env fields safely and load them here, or change the recovery contract so `AgentManager` passes the known `RegisterAgentConfig.env` into the adapter and the adapter spawns with that exact env.

- **runtime/agent-runtime/pty/claude-pty.ts:171-180 — Fail-closed crash classification leaves the PTY alive and unrestarted** [high]
  When parsing returns `unknown`, the adapter emits `crashed` and writes a crash marker, but it does not kill the PTY, mark `state.alive = false`, or remove the handle. Since `isAlive` continues to return true for this state, the heartbeat has no immediate liveness signal to restart it; the manager's `crashed` callback path only cascades children. A real parser drift or unrecognized long output can therefore strand the handle in a crashed state while the process remains writable and recovery is deferred until a later stall or daemon reboot.
  - **Action:** Make the fail-closed path terminal: stop accepting sends, kill the PTY or mark it not alive, and route through a restartable manager/heartbeat path. Add a test that unknown output causes `isAlive` to become false or triggers a shutdown/restart, not just a status callback and marker write.

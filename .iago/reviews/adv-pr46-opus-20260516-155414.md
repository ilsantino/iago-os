### [CRITICAL] runtime/integration/hello-world.test.ts:182-296 — Hello-world "end-to-end" test never wires the real daemon
**Why:** Acceptance criterion #3 ("End-to-end integration test: register Claude PTY agent → claim task → Telegram approval → resume — runnable via single command") explicitly requires a wired-daemon path. The actual test (`it("file-bus claim, approval roundtrip, and resolved-output flow")`) does NOT call `startDaemon()` at all — it directly invokes `claimTask`, `createApprovalRequest`, `resolveApproval`, `waitForApproval`, `writeResolvedOutput` as plain library calls. The Telegram mock's `telegramCallbackHandlers` array is populated to nothing (no `bot.on("callback_query", ...)` is ever exercised because `bot.start()` is never called in this `it`). The "simulate a Telegram allow callback" step at line 235-241 is just `resolveApproval(approvalId, "allow", "test-user")` directly — the same library call the bot would make, but with no bot in the loop. This is a unit-composition test, not an integration test. The acceptance criterion is NOT met.

Plan 07 Task 3 step 7 is explicit: "simulate a Telegram callback `approve_allow_<id>` — invoke the bot's mocked callback handler directly." The shipped test bypasses the callback handler entirely.

The second `it` ("daemon startup wires components and shuts down cleanly") DOES call `startDaemon()` but with `agents: []` and `telegram: null` — it exercises lifecycle only, asserts `listHandles() === []`. No agent is registered, no task is claimed through the daemon, no approval is requested through the bot, no PTY adapter is exercised end-to-end. The agent-runtime registry side-effect import path (`import("../agent-runtime/pty/claude-pty.js")` at main.ts:88) is not asserted to have populated the registry.

The third `it` (telemetry) only asserts `daemon-start` + `daemon-stop` kinds — NOT the 6 canonical events the plan demands (Plan 07 Task 3 step 12: "daemon-start, agent-registered, agent-spawned, task-claimed, approval-requested, approval-resolved").

**Fix:** Rewrite the `it("file-bus claim, ...")` test to:
1. `startDaemon({ telegram: { token: "test-token", allowedUserIds: [42] }, agents: [{ agentId: "claude-main", runtimeId: "claude-pty", cwd: ..., env: {...}, autoStart: true }], ... })`.
2. Assert `daemon.agentManager.listHandles().length === 1`.
3. Write the task to `tasks/pending/<id>.json` via `claimTask` AS the agent (not as the test).
4. Invoke the mocked bot's callback handler from `telegramCallbackHandlers[0]({ id: "q1", data: "approve_allow_<id>", from: { id: 42 } })`.
5. Assert `telegramSentMessages` contains the approval-broadcast message (proves bot sent it).
6. Assert ALL 6 canonical telemetry events present in NDJSON.
7. Drop the test-direct calls to `resolveApproval`/`waitForApproval` — those must fire through the bot's `on("callback_query")` handler.

### [IMPORTANT] runtime/daemon/main.ts:107-123 — `bootRecovery()` called WITHOUT knownConfigs at the production call site
**Why:** Codex flagged this as HIGH. Confirming the placement: line 109 reads `const bootResult = await agentManager.bootRecovery();` — no opts argument. agent-manager.ts:474 reads `const knownConfigs = opts?.knownConfigs;` and subsequent branches (lines 492, 535) check `if (knownConfigs !== undefined)` — so on the production startup path, the daemon-crash-without-marker recovery branch (the one daemon/README.md:36-41 explicitly promises handles the "Absent marker on next boot ... daemon itself crashed" case) is NEVER exercised. The recovery code is dead on production startup. README.md:39-41 is honest: "without that map the case cannot fire because the daemon has no surviving record of which handles to recover." So the fix is also documented missing: nothing yet persists the surviving handle config registry across restarts for main.ts to load. Phase 1 ships a daemon that, if it crashes, leaves all formerly-registered agents permanently stranded.

**Fix:** Plan 07 main.ts MUST load the persisted `<state-root>/agents/<handleId>.json` files (already written by `AgentManager.persistAgent`) into a `Map<handleId, RegisterAgentConfig>` BEFORE calling `bootRecovery`, then pass `{ knownConfigs }`. The directory and file format already exist (agent-manager-write-side). Add an integration test: write two `agents/<id>.json` files to a tmp state root, no markers, call `startDaemon()`, assert both handle ids appear in `bootResult.crashes`. This finding is a follow-on to Codex's report — the actual integration test in this PR does NOT cover this regression surface, so even if Codex's fix lands it will not have a test guard.

### [IMPORTANT] runtime/migration/phase-1-rollback.md:111-117 — Step 6 ("Re-run Phase 1 after rollback") is wrong if PRs merged
**Why:** Step 6 instructs `git checkout feat/v2-runtime-skeleton-agent-runtime-interface` (the Plan 01 branch). ### [IMPORTANT] runtime/migration/phase-1-rollback.md:14-30 — SIGTERM hang behavior not bounded
**Why:** The doc says "If it does not exit within 30s, force-kill" but the `kill -KILL <pid>` step is presented as a backup, not a guarantee. The daemon's own shutdown path (main.ts:162-206) has NO outer timeout — `heartbeat.stop()`, `bot.stop()`, `ipcServer.stop()`, and the per-handle `shutdownAgent` loop are all awaited serially with no `Promise.race(timeout)`. If a single agent's adapter `shutdown` hangs (plan 04 stress-test scenario), the daemon never exits. The rollback runbook does not document how long to wait before SIGKILL — "30s" is asserted but unmotivated and not enforced by the daemon code. On Windows the PowerShell step (`Stop-Process`) is the equivalent of SIGKILL, NOT SIGTERM — so the Windows path skips the graceful shutdown entirely and the `.daemon-stop` markers are NOT written.

**Fix:** Bound the per-stage shutdown in main.ts with `Promise.race([stage, sleep(10_000)])` per stage (heartbeat, bot, ipc, each handle). Update the rollback doc to document the cumulative timeout budget. On Windows, document a graceful path (e.g., send Ctrl+C via `taskkill /pid <pid>` without `/F` first, fall back to `Stop-Process` only on timeout).

# Codex Adversarial Review

Target: branch diff against ecfb29932daf95e01066fa86c9f3fab87834eaa0
Verdict: needs-attention

No-ship: the daemon wiring can either load zero PR-triage crons in production or turn cron files into resolved/no-op tasks without ever running the agent.

Findings:
- [high] Production daemon looks for crons under dist/agents, so PR-triage is silently not registered (runtime/daemon/main.ts:353-357)
  `resolveAgentsDir()` derives the agents directory from `import.meta.url` and returns `../agents`. The production unit runs `/opt/iago-os/runtime/dist/daemon/main.js`, and `runtime/tsconfig.json` only compiles TS into `runtime/dist`; it does not copy `runtime/agents`. In that runtime, this resolves to `/opt/iago-os/runtime/dist/agents`, and `loadCronEntries()` treats ENOENT as an empty cron list. The daemon still starts, but no PR-triage schedule is registered, so the daily workflow never fires.
  Recommendation: Resolve the default agents directory to the repo `runtime/agents` path used by the deployed checkout, or package/copy agent assets into `dist/agents`; add a production-path test that simulates `dist/daemon/main.js` and fails when expected cron entries are absent.
- [high] Cron-fired task files are never dispatched to claude-pty (runtime/daemon/main.ts:1100-1102)
  The new startup path starts `scheduler.start()` and `agentManager.startPollingLoop()`, but the polling loop only claims files. Existing `AgentManager.claimTask()` is explicitly decrement-only and does not dispatch task content to a runtime, and `isAgentRegistered()` only checks live handles. Since `startDaemon()` does not discover `runtime/agents/pr-triage/agent-config.json` and that file has `autoStart: false`, a cron file will either stay pending as `task-unrouted`, or, if a live handle happens to exist, be moved to resolved without running the prompt. That loses the daily Telegram message and can create false `task-resolved` telemetry.
  Recommendation: Wire the task path to actual agent execution: load the agent config, spawn or route the prompt through `claude-pty`, and move/emit terminal state only after execution completes or fails. Cover it with an integration test from cron tick to runtime invocation.

Next steps:

- Add real dispatch semantics for cron task files before treating PR-triage as wired.

# Codex Adversarial Review

Target: branch diff against main
Verdict: needs-attention

No ship: the new shutdown timeout override is accepted without safety bounds, so a test affordance can make daemon shutdown report success while skipping real teardown.

Findings:
- [high] Unvalidated shutdown timeout can turn graceful shutdown into an immediate best-effort skip (runtime/daemon/main.ts:335-336)
  `stageTimeoutMs` is taken directly from `config.shutdownStageTimeoutMs` and passed to every `withTimeout` call. Because `withTimeout` uses `setTimeout(..., timeoutMs)`, a programmatic caller can pass `0`, a negative number, `NaN`, or an unrealistically tiny value and force heartbeat, IPC, bot, and per-agent shutdown stages to time out immediately. The daemon then still emits `daemon-stop` and resolves shutdown, while the underlying stop/shutdown promises may still be pending and may not complete before process exit. With live agents, that can leave adapter subprocesses alive or prevent graceful marker/telemetry work from completing, while observability says the daemon stopped cleanly. This is an inference from the new public `DaemonConfig` field and the direct use at shutdown.
  Recommendation: Validate `shutdownStageTimeoutMs` before constructing `shutdown`: require a finite positive integer and enforce a sane minimum, or keep the short timeout as a private test-only injection path that cannot be supplied through the exported daemon config shape. Add tests for rejected `0`, negative, `NaN`, and too-small values.

Next steps:
- Add validation/clamping for `shutdownStageTimeoutMs` and cover invalid values in tests.
- Consider asserting that timeout-triggered shutdown with live handles does not emit misleading clean telemetry before required teardown invariants are satisfied.

## Notes

The PR body artifact on disk (`_pr-body-b03-main.md`) under-claims the scope; verify the GitHub PR body matches before approving.

The atomic-rename two-variant API (`atomicRename` strict vs `atomicRenameStaleDest`) introduced in Plan 02 is the load-bearing primitive for everything downstream — the strict variant is now what makes the approval-bus CLAIM step safe. Future callers must classify themselves (race / stale-dest / collision-hazard) per the `state-paths.md` audit; the audit is the contract.

If a future refactor reorders boot, the bot's first poll cycle could observe stranded inflight as "already-resolved" and silently drop a user decision. Worth pinning down in an integration test that asserts the call ordering rather than relying on code-order alone (current `main.test.ts` covers the helpers but doesn't assert the ordering invariant).

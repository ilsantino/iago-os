# Plan 01 — Flatten Ayuda UX to sidebar layout

**Feature slug:** feature-ayuda-chat (plan 01)
**Branch:** `feat/ayuda-sidebar-flatten`
**PR base:** `sentria-qc`
**Stack:** React 19 + Vite + Tailwind v4 + react-router v7
**Test posture:** manual (sandbox + browser) — no automated suite per repo CLAUDE.md
**Depends on:** **PR #121 (`feat/onboarding-usage-docs`) MERGED to `sentria-qc`.** The files this plan modifies (`src/pages/AyudaPage.tsx`, `AyudaDocPage.tsx`, `AyudaAudiencePage.tsx`) only exist on that branch. Do NOT start pipeline until PR #121 lands.

## Context

PR #121 shipped a 3-tier Ayuda navigation: cards grid (`/ayuda`) → audience index (`/ayuda/:audience`) → doc page (`/ayuda/:audience/:slug`). Visual user smoke test (2026-05-15) showed the flow works but feels heavy — users click three times before reading a doc, and the audience-index page adds little.

**Goal:** Reduce to one click from sidebar `Ayuda` link to a useful doc page. Sidebar-layout pattern (Notion, GitBook, Stripe docs).

**Behavioral change:**

| | Before (PR #121) | After (this plan) |
|---|---|---|
| `/ayuda` lands on | Card grid | Sidebar + default doc rendered (conceptos) |
| Click to reach a specific doc | 2 clicks (card → slug) | 1 click (sidebar item) |
| URL for a specific doc | `/ayuda/:audience/:slug` | unchanged |
| URL for an audience root | `/ayuda/:audience` (audience-index page) | redirects to `/ayuda/:audience/${firstSlug}` |
| Mobile | Card grid scrolls | Sidebar collapses to sheet (button toggles) |

Only TWO URL shapes after this plan: `/ayuda` and `/ayuda/:audience/:slug`. The intermediate `/ayuda/:audience` becomes a redirect-only route — no placeholder UI, no audience-index page.

## Out of scope

- LLM chat (Plan 02 of this feature).
- Search inside docs (deferred — could ship with Plan 02 if scope allows).
- Reorganizing the docs themselves — keep all 24 files unchanged.
- Changing AppRole → Audience mapping (`AUDIENCE_BY_ROLE` in current AyudaPage stays as-is).
- Server-rendered docs (Vite glob + react-markdown at runtime continues).

## Tasks

### T1 — Replace AyudaPage with sidebar layout

**File:** `src/pages/AyudaPage.tsx` (rewrite entirely).

Layout: two-column desktop, single-column with sheet on mobile.

- **Left pane (sidebar)**: 240–280px fixed width on desktop.
  - Header: "Ayuda" with `HelpCircle` icon.
  - Audience groups (one per visible `Audience` from `AUDIENCE_BY_ROLE[role]`):
    - Group header = audience title (e.g. "Administradores") with chevron-down icon.
    - Groups are always expanded on mount (no collapse-state persistence in v1).
    - Group body = `<NavLink>` per doc slug (NOT plain `<Link>` — `NavLink` provides built-in `isActive` styling). Active doc gets `bg-muted` + `font-medium`.
    - Use ShadCN `Collapsible` primitive for group toggles — collapse state is session-only React state, not persisted.
  - Footer: short line: "¿Necesitas más ayuda? Escríbenos al equipo de Sentria."
- **Right pane (content)**: takes remaining width.
  - Renders the markdown doc resolved from `:audience/:slug` URL params via the same Vite `import.meta.glob` + `react-markdown` pattern from current `AyudaDocPage`.
  - **DO NOT add `rehype-raw` / `skipHtml={false}` / `allowDangerousHtml`** to the renderer. Markdown only, no raw HTML.
  - Same prose-style arbitrary-descendant Tailwind classes as `AyudaDocPage` line 98.
  - When the URL is `/ayuda` (no audience/slug), default to `audience=conceptos, slug=conceptos`.
  - When the URL has a slug that doesn't resolve in `docModules`, render a "Guía no encontrada" placeholder (not blank, not crash): heading + short line + back-link to `/ayuda`.
  - `h1` focus on doc change (a11y, matches current behavior). Use `useRef` + `useEffect` on `slug` change.
- **Mobile (< md breakpoint, < 768px)**: sidebar hidden, button in top-right of content pane labeled "Guías" with menu icon. Button is **sticky** (`sticky top-0`) so it stays visible on long docs. Click opens a `<Sheet>` from `@/components/ui/sheet` (confirmed to exist — no fallback to `Dialog` needed).

**Sidebar navigation:**
- Use `<NavLink>` with `to={...}` for each doc.
- Use `navigate(..., { replace: true })` for intra-sidebar navigation (programmatic, e.g. on collapse toggle if needed). This prevents the back-button from accumulating 24 history entries from casual browsing.
- The URL updates on every doc click. Doc content reactively renders via `useParams`.

**Auth-loading race:**
- When `useAuth()` returns `loading=true` OR `user?.appRole` is still `undefined`, render a loading skeleton in the sidebar (skeleton group placeholders) so the audience list doesn't flicker. Match the current AyudaPage's skeleton pattern. Right pane shows a skeleton block during this state.

### T3 — Consolidate routes in `src/App.tsx`

After T2, routes are:
- `/ayuda` → `<AyudaPage />` (sidebar + conceptos default in right pane)
- `/ayuda/:audience` → `<AyudaAudienceRedirect />` (thin redirect to the first slug in that audience; uses `<Navigate replace />`)
- `/ayuda/:audience/:slug` → `<AyudaPage />` (sidebar + selected doc in right pane)

Two visible URL shapes (`/ayuda`, `/ayuda/:audience/:slug`); `/ayuda/:audience` exists only as a redirect for shareable / bookmarked links.

**`<AyudaAudienceRedirect />`** is a tiny inline component:
```tsx
import { Navigate, useParams } from 'react-router-dom'
import { listDocsForAudience } from '@/lib/ayuda'

function AyudaAudienceRedirect() {
  const { audience } = useParams<{ audience: string }>()
  const docs = audience ? listDocsForAudience(audience) : []
  const firstSlug = docs[0]?.slug
  if (!firstSlug) return <Navigate to="/ayuda" replace />
  return <Navigate to={`/ayuda/${audience}/${firstSlug}`} replace />
}
```

Remove the imports of `AyudaAudiencePage` and `AyudaDocPage` from `App.tsx`.

### T4 — Slug → title resolver + audience doc lister

**File:** `src/lib/ayuda.ts` (new). Moves the helpers currently in `AyudaAudiencePage.tsx`.

**Return-type contracts (explicit):**

```ts
export interface AyudaDoc {
  audience: string
  slug: string
  title: string
}

/** Strip "NN-" prefix, replace dashes with spaces, capitalize first letter. */
export function slugToTitle(slug: string): string

/** Returns docs for the given audience, sorted alphabetically by slug. */
export function listDocsForAudience(audience: string): AyudaDoc[]

/** Returns ALL docs across all audiences (for Plan 02 chat corpus). */
export function listAllDocs(): AyudaDoc[]
```

The `listAllDocs()` function is added now so Plan 02 (chat) doesn't need a breaking refactor.

`AyudaPage.tsx` consumes `listDocsForAudience(audience)` to build the sidebar.

### T5 — Mobile sheet

On `< md` (< 768px), the sidebar disappears and a **sticky** "Guías" button shows in the right pane's header. Clicking opens a `<Sheet>` from the left with the same sidebar contents.

**Auto-close on navigation:** the Sheet's `open` state is tied to a local `useState`. A `useEffect` listens to `location.pathname` — when the path changes, set `open=false`. This ensures tapping a doc link inside the sheet closes the drawer and navigates atomically.

```tsx
const [sheetOpen, setSheetOpen] = useState(false)
const location = useLocation()
useEffect(() => { setSheetOpen(false) }, [location.pathname])
```

Tapping a NavLink inside the Sheet triggers route change → Sheet closes via the effect. No manual close-handlers per link.

### T6 — Accessibility + minors

- `prefers-reduced-motion: reduce` → all transitions use `motion-safe:` variants (established convention).
- Sidebar NavLink items rely on `NavLink`'s built-in `aria-current="page"` when active. No manual wiring.
- H1 inside content pane gets `tabIndex={-1}` and a focus call on slug change (existing pattern).
- Sidebar group chevrons rotate on expand/collapse via CSS transform, motion-safe.
- Color contrast: active doc NavLink uses `bg-muted` + `text-foreground`; inactive uses `text-muted-foreground hover:text-foreground hover:bg-muted/50`.
- "Guía no encontrada" placeholder is reachable by keyboard (tabIndex on the back-link).
- The mobile "Guías" button has `aria-label="Abrir menú de guías"` for screen readers.

### T7 — Inline notes for security posture

Add a brief comment block near the top of `src/pages/AyudaPage.tsx` (or in `src/lib/ayuda.ts`) documenting two invariants for future maintainers:

```tsx
// Security posture:
// 1. The sidebar role-filter (AUDIENCE_BY_ROLE) is UX-only, NOT an access control.
//    A reporter can deep-link to /ayuda/admin/05-vincular-telegram and the doc
//    will render — the route resolver does not gate by role. Treat all 24 docs
//    as addressable by any signed-in user. Do NOT embed admin-only operational
//    secrets (AWS account IDs, internal IPs, support-tool credentials) in
//    docs/usuarios/ content.
// 2. The markdown renderer MUST NOT use rehype-raw / skipHtml={false} /
//    allowDangerousHtml. Raw HTML is a real XSS surface, especially once
//    Plan 02 (LLM chat) streams model output through the same renderer.
```

This is a comment, not a runtime check. The intent is to surface the constraint in code review for any future change that might violate it.

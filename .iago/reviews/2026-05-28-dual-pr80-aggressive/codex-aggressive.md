# Codex Adversarial Review

Target: branch diff against 94582e3603faa39671651176d64338b491b86f19
Verdict: needs-attention

No ship: the dispatch path can duplicate or silently lose work under ordinary timing races.

Findings:
- [high] Pending task can be dispatched multiple times before it is claimed (runtime/daemon/agent-manager.ts:1678-1690)
  `processPendingTask` emits `task-dispatch-needed` and returns while the file remains in `tasks/pending/`; the real listener is detached with `void taskDispatchHandler(evt)`, so the polling loop does not wait for `runtime.send` or `claimTask`. If `runtime.send` takes longer than one polling interval, the next tick rereads the same file and emits another dispatch for the same persistent PTY. The first handler may eventually claim the file while the second has already sent the same prompt, producing duplicate PR triage actions and confusing terminal telemetry.
  Recommendation: Add a per-filename in-flight guard or atomically move the task to an inflight/claimed directory before calling the async dispatch handler; only retry from a durable failure state, not while a send is in progress.
- [high] Shutdown removes the dispatcher before stopping the poller, enabling silent resolve without dispatch (runtime/daemon/main.ts:1303-1317)
  Shutdown calls `removeAllListeners("task-dispatch-needed")` before `stopPollingLoop()` clears the interval. A tick that starts in that gap sees `listenerCount(...) === 0` and takes the backwards-compatible fallback to `claimTask`, moving a pending task to `tasks/resolved/` without ever sending it to the runtime. This is a real data-loss path during deploy/restart because the resolved event also releases the cron slot and makes the task look completed.
  Recommendation: Stop and drain the polling loop before removing the dispatch listener, or remove the listener-less `claimTask` fallback once the daemon wires dispatch unconditionally.
- [medium] Malformed task payloads are acknowledged after sending an empty prompt (runtime/daemon/main.ts:593-613)
  The handler converts any non-string `taskContent.prompt` to `""`, sends that to the runtime, and then claims the task. A corrupt or partially written task with a valid `agentId` but missing `prompt` will be marked resolved even though the intended work was never delivered. This violates the poison/retry behavior used for bad task files elsewhere and makes upstream schema drift hard to detect.
  Recommendation: Validate `prompt` as a non-empty string before dispatch; poison the task or emit `pr-triage-dispatch-failed` without claiming so the failure remains recoverable and visible.

Next steps:

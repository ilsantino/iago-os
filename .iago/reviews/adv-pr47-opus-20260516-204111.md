### I4 — Operational edge case: cutover-script fires while Santiago's laptop sleeps or Tailscale node disconnects

**Files:** `03-cutover-rollback-orchestration.md` (cutover.sh task 1, rollback.sh task 2)

Plan 03 stress I2 covers `IAGO_CUTOVER_RESUME_FROM=Tnn` resumability "when a cutover is interrupted mid-flight (e.g., Santiago's laptop dies between T+07 and T+10)". But the practical failure modes broader than laptop-death:

- **Tailscale node disconnects mid-`tailscale ssh ... -- systemctl ...`** — bash `set -e` exits non-zero; resume mechanism activates, but the prior step may have half-completed (systemd-creds was uploaded, daemon not yet enabled). Need an idempotent re-run guarantee per step.
- **Windows laptop sleeps during the 30-min stay-at-keyboard monitoring window** — `cutover.sh` exited cleanly at T+60, no impact. But the post-cutover monitoring depends on Santiago seeing journalctl + Telegram on his phone; documented but not enforced.
- **OpenClaw is mid-task during stop in `archive-openclaw.sh` step 1** — script greps `pgrep -u ilsantino -f openclaw-gateway` and exits 1 if non-empty. Plan 02 stress doesn't cover the "task in flight when stop fires" semantics: OpenClaw user systemd unit stop sends SIGTERM, daemon's task in flight is interrupted. Acceptable per FAST cutover, but the spec/plan stack doesn't say so explicitly.
- **BotFather rate-limit on `/revoke` during cutover** — spec § 3 atomic sequence assumes /revoke succeeds first try. Plan 02 stress M2 acknowledges propagation delay but not rate-limit (BotFather limits /revoke to ~1/min per bot per Telegram docs).

**Required fix:** Add to Plan 03 Task 1 cutover.sh an explicit idempotency guarantee per step (each step has a pre-condition check that lets the same step re-run cleanly without partial state). Add to Plan 02 Task 4 rotate-telegram-bot.sh a retry on /revoke failure with backoff (separate from the propagation-delay retry already noted). Document the OpenClaw-task-in-flight semantics in Plan 02 Task 1 action paragraph.

### I7 — age pubkey corruption silent acceptance

**File:** `02-openclaw-teardown-scripts.md:104` (Plan 02 stress-test C3)

Plan 02 stress C3 caught this: *"Add a simple check: `head -c 21 $ENCRYPTED_PATH | grep -q '^age-encryption.org/v1'` ... catches the obvious failure mode."* The forward-list (line 132) item 3 confirms the fix is required.

But the *kind* of corruption the magic-byte check catches is narrow: a pubkey that age accepted-as-valid but produced wrong-key ciphertext IS still age-encryption.org/v1 format.

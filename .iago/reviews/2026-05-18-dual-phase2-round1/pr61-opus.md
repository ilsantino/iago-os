## Important

- **I1 — Same-second filename collision under `maxConcurrent > 1`.** `cron-scheduler.ts:552-553` derives the task filename as `<prefix>__<unix>.json` where `unix = Math.floor(now.getTime() / 1000)` (seconds precision). Combined with the registration-time uniqueness check on `outputTaskNamePrefix`, the production 60s-tick guarantees ticks are minutes apart, so collision is impossible. BUT: any caller that registers with `maxConcurrent > 1` and dispatches two fires within the same wall-clock second (e.g., via `_tickForTests` in tests, or future "fan-out" usage where multiple matches land in one tick batch) lands at the same `finalPath`, and `atomicRenameStaleDest` silently unlinks-then-renames on EEXIST/EPERM — overwriting the first task file. The registration check explicitly says "prefixes must be unique to prevent task-file collisions" but only covers the cross-cron case, not the within-cron same-second case. **Recommendation:** append a small disambiguator (counter or pid+random suffix like the tmp file already does — `<prefix>__<unix>__<pid>-<rand>.json`) OR upgrade `unix` to millisecond precision (`Math.floor(now.getTime())`). Test (14) at `cron-scheduler.test.ts:691` proves the issue exists today: it has to *manually* advance `nowHolder` by 30s between two `_tickForTests` calls to avoid filename collision — that's the bug surfacing as a test workaround. The same workaround will be required by every future test or production fan-out path. Severity is Important not Critical because Plan 04b's only documented consumer is `pr-triage` at `maxConcurrent: 1` on a daily 60s-spaced cron — but any future `maxConcurrent > 1` consumer will silently lose tasks.

## Minor

- **M1 — `as` casts present in test file despite plan-wide prohibition.** Plan Task 1 says "NO `any`, NO `as` casts (use Zod-or-handwritten type guards for the JSON parse)" — the impl honors this cleanly via `toFiveTuple` at `cron-scheduler.ts:151`. But the test file ships four `as` casts: `cron-scheduler.test.ts:34` (`null as ((e: DaemonEvent) => Promise<void>) | null`), `:63` (`JSON.parse(line) as Record<string, unknown>`), `:456` (`pending[0] as string`), `:459` (`JSON.parse(body) as Record<string, unknown>`). CLAUDE.md "TypeScript strict — no `as` casts (except type guards)" applies to all TS, not just impl. Lines 63 and 459 are JSON-parse narrowings — Plan Task 1 explicitly called these out as needing type guards. Replace with the same `toFiveTuple`-style guard pattern or a `parseJsonRecord(line)` helper. Lines 34 and 456 are nullable-narrowing — can use `if (x === undefined) throw …` or `assert`.

- **M2 — `maxConcurrent: 0` silently coerced to 1.** `cron-scheduler.ts:367-370`: `const maxConcurrent = typeof opts.maxConcurrent === "number" && opts.maxConcurrent > 0 ? Math.floor(opts.maxConcurrent) : 1;`. A caller intending "never fire this cron" (`maxConcurrent: 0`) gets unlimited firing-up-to-1 silently. Same for negative values. Reject at `registerCron` with `TypeError` so the misconfiguration surfaces loudly. (Plan said default 1 if omitted, but didn't specify "silent coerce on bad input"; fail-loud is the Garry-impressed standard.)

- **M3 — `promptTemplatePath` not validated.** `cron-scheduler.ts:540` reads `fs.readFileSync(cron.promptTemplatePath, "utf8")` with no path-sanity check. `agentId` and `outputTaskNamePrefix` go through `assertSafeIdentifier`; `promptTemplatePath` does not. Phase 2 daemon is single-tenant (Santiago's VPS) so practical risk is low, but defense-in-depth would reject paths with `..` traversal or NUL bytes. The error path (ENOENT) is handled gracefully via `cron-fired-prompt-missing`, so this is a hardening note not a correctness bug.

- **M4 — Test API leaks at runtime.** `_tickForTests` (`cron-scheduler.ts:422`) and `_runningCountForTests` (`:429`) are JSDoc'd `@internal` but are real public methods on the class. Anyone with a `CronScheduler` reference can drive ticks directly or inspect internal state. JSDoc `@internal` is documentation, not enforcement. For a daemon class with a 60s tick contract, allowing external callers to fire ticks at will breaks invariants the documented API protects (e.g., the documented contract "after `await stop()` returns, no more `cron-fired` events"). Acceptable for Phase 2 because the daemon process owns the only reference, but flag for Phase 3+ when the class might surface in other reach paths.

- **M5 — No fsync on tmp before rename.** `cron-scheduler.ts:569` uses `fs.writeFileSync(tmpPath, body, { encoding: "utf8" })` then `atomicRenameStaleDest`. Compare `file-bus.ts:158-169` which uses `fs.promises.open(claimPath, "wx") → writeFile → datasync → close`. If the daemon crashes between writeFileSync and the kernel flushing the tmp page-cache, the rename succeeds but the file content is zero or partial on next boot. For 200-byte task files this window is microseconds and the file-bus pattern would protect against it. Task files are recoverable (next cron tick re-fires), so this is "less durable than Phase 1 sibling code" rather than "data loss" — but the inconsistency with file-bus.ts is jarring. Either: (a) document the durability tradeoff in cron-scheduler.README.md § 6, or (b) match the file-bus pattern (`fsp.open + writeFile + datasync + close`).

- **M6 — Silent overlap-skip with no telemetry.** `runTickGuarded` (`cron-scheduler.ts:438`) short-circuits when `tickInFlight !== null` and emits nothing. If a wake-check blocks for 25s and the next setInterval tick fires, that tick is silently dropped. Operators can't distinguish "this minute had no matching cron" from "this minute was skipped because last tick was still running". Add a `cron-tick-overlapped` telemetry kind (or reuse `cron-overlap-prevented` with a sentinel `agentId: '__tick__'`) so production drift is observable. Currently impossible to debug from telemetry alone.

- **M7 — Constructor signature diverges from plan.** Plan Task 1: `constructor(opts: { agentManager: AgentManager; fileBus: FileBus; stateRoot: string; logger?: Logger })`. Impl: `agentManager: CronAgentManager (EventEmitter); stateRoot?: string; logger?: Logger; nowFn?: () => Date` — no `fileBus`, `stateRoot` made optional, `nowFn` added. The functional outcome is equivalent (uses `pathFor("tasks/pending")` directly), and `CronAgentManager = EventEmitter` is a cleaner cross-plan coupling than typing against the not-yet-existing AgentManager-with-task-resolved-emit shape. But the plan explicitly listed `fileBus: FileBus` and `stateRoot: string` — Plan 04b consumers reading the plan will write `new CronScheduler({ agentManager, fileBus, stateRoot: ... })` and TypeScript will reject the extra `fileBus` field with strict object literal checks. Mention the divergence in the README § 2 wiring example (which already does — line 102 shows only `{ agentManager }`) AND in plan 07a's "implementer forward-list" so Plan 04b authors don't get tripped up.
